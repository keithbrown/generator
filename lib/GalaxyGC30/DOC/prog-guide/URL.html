URL



The URL class (vurl) encapsulates URLs to make them easy to use in a Galaxy program. Several other classes including vbrowse and vhelpref use vurl. Refer to the Web Browse and Help chapters for more information on those classes use of URLs.

The URL class provides ways of encoding and decoding URL strings, operating on the component parts of the URL, and resolving URLs relative to a base URL. The algorithms used to parse URLs are based on the specifications in RFC1808.

A URL is made up of several components. Assume that you have the following URL:

    http://www.a.com/b/c/d;p?q#f
The portions of the URL have the following names:

    Scheme The scheme is the portion before the first colon (":"). (http)

    Net Location The net location is the portion after the first double slash ("//") up to the next slash ("/") or the end of the string. (www.a.com)

    Path The path is the central portion of the URL. (/b/c/d)

    Parameters The parameters are the portion from the first semi-colon (";") character to the end of the string. (p)

    Query The query portion of the URL is after the first question mark ("?") to the end of the string. (q)

    Fragment The fragment of the URL is the portion after the first "#" character. (f)

Key Functionality

 You can insert an entire URL path by passing a null terminated vchar string to FromString. For example:

    vurl *myURL = vurlCreate();
vurlFromString(myUrl,"http://a/b/c/d;p?q#f");
The different portions of the URL are parsed out of the string. Note that they are parsed out in the following order:

  1. Fragment
  2. Scheme
  3. Net Location
  4. Query
  5. Parameters
  6. Path
The different parts of the URL, however, can be manipulated using the appropriate Get* and Set* functions. For example, to set the scheme portion of the URL use SetScheme or SetSchemeScribed. To get a vchar string representation of the fragment, use GetFragment.

Use Resolve to resolve a URL relative to some other URL. A relative URL is one which does not have a scheme but which may contain either a relative or absolute path. Resolution of a relative URL involves the following steps:

  1. The relative URL inherits the scheme of the base.
  2. The relative URL inherits the net location of the base, if it does not already have one.
  3. The path is resolved:
    1. It the relative URL's path is empty, it inherits the path of the base. If the relative URL does not have a parameters component or a query component, each is inherited from the respective base components.
    2. If the relative URL's path does not begin with a slash "/", it is considered relative. The base URL's path is prepended (minus any terminating file element) to that of the relative URL.
    3. If the relative URL's path begins with a slash "/", it is considered absolute and is not affected by the base path.
Relative paths may contain ".." elements to effectively remove elements from the base path during resolution. See Section 4 of RFC1808 for additional information concerning this algorithm.
