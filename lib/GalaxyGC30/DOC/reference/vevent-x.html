vevent

        typedef ... vevent;
Notification of some operating system or window system event of interest. A vevent can also be created and posted by the application itself.

A vevent contains information about the event that it is reporting. A vevent is assigned a time when the event is to be dispatched, an event type, an optional event binding (a vname *), an event priority, an optional target handler procedure (a veventHandlerProc), a synchronization flag, and information specific to the event type.

Events from the operating system and the windowing system are automatically created and queued by the Galaxy event loop (veventProcess). In addition, events can be created by the client and posted to the queue immediately or into the future.

veventCancelWaitProc

        typedef ... veventCancelWaitProc;
A procedure designed to be called from interrupt code or from code in a thread separate from the event loop. The purpose of this procedure is to unblock a blocked event loop. The current veventCancelWaitProc is called by the various veventPostAtInterrupt calls.

veventClientEvent

        typedef ... veventClientEvent;
The client event class allows you to create and dispatch events for your own use. You should set the Target callback attribute (with veventSetTarget to tell the Event Manager what to do with the event when it is dispatched. You can also set the ClientData and ClientType attributes to identify particular events.

veventClientQueueFlushProc

        typedef ... veventClientQueueFlushProc;
Reads events from the window system event queue and converts them to type vevent. This function is necessary for window systems that bypass the normal file stream of event data, but rarely needs to be used by a client. One possible use is when the client is receiving events from two or more different sources.

veventClientWaitIndefinitelyProc

        typedef ... veventClientWaitIndefinitelyProc;
This procedure gets called by the event loop to block for an indefinite amount of time. This happens when there is not a future vevent, and when there are no pending messages in the operating system or window system event queues. An indefinite wait is cancelled by either the receipt of an operating system or window system event or by having called the current veventCancelWaitProc.

veventClientWaitProc

        typedef ... veventClientWaitProc;
This procedure gets called when the event loop is about to block for a finite amount of time. This happens when there is a future vevent in the Galaxy queue, and when there are no pending messages in the Galaxy, oeprating system, or window system queues. An indefinite wait is cancelled by either the receipt of an operating system or window system event or by having called the current veventCancelWaitProc.

veventDispatcherProc

        typedef ... veventDispatcherProc;
A procedure used to dispatch events to the appropriate veventHandlerProc. A new dispatcher procedure could be used for journaling. All events are dispatched through the current veventDispatcherProc. Events with a target (see veventSetTarget) are usually sent directly to the targeted veventHandlerProc. All other events are usually sent to the current veventHandler (compare with veventGetHandler).

Example: change the dispatcher to enable event journaling. 

static FILE *recorderFile;
veventDispatcherProc oldDispatcher;

int newDispatcher(vevent *event, const veventHandler *handler)
{
    veventDump(event, recorderFile);
    return (*oldDispatcher) (event, handler);
}

int main (int argc, char *argv[])
{
    recorderFile = fopen("recorder", "w");

    ...

    oldDispatcher = veventRegisterDispatcher(newDispatcher);
    veventProcess();

    fclose(recorderFile);
}

veventFD

        typedef ... veventFD;
A veventFD holds information used by the file descriptor translation procedures. You can obtain a veventFD pointer from a file descriptor (such as returned from the system call open) by calling veventRegisterFD. You can then set read, write, or exception notification on the veventFD.

veventFD only works on UNIX platforms.

veventHandlerProc

        typedef ... veventHandlerProc;
A procedure used to handle an event. The handler procedure must return vTRUE if the event was handled and vFALSE if it was not. An event's target handler procedure is set through veventSetTarget. An event handler's handler procedure for a certain event type can be set through veventSetHandlerProc. The handler does not get ownership of the event and should not destroy the event. It may, however, repost the event using veventPost or any of the other post calls.

veventPreBlockHandlerProc

        typedef ... veventPreBlockHandlerProc;
A procedure that is called just before the event loop is about to block for a non-zero amount of time. The vevent module will call the current preblock handler (gotten via veventGetPreBlockHandlerProc) just before it blocks. Client code should also call the current veventPreBlockHandlerProc (via veventGetPreBlockHandlerProc) just before any blocking call or even any long calculation (greater than a second).

This may not be used to cancel the pending blockage or stop this program from being blocked. This is also not useful for a program that wants to perform idle time processing. Use a veventClientWaitProc or veventClientWaitIndefinitelyProc for that.

veventPriority

        typedef ... veventPriority;
An integral type representing the importance of the event. Events in the queue with a higher priority are dispatched before events with a lower priority. Values should be between veventPRIORITY_MIN and veventPRIORITY_MAX, inclusive. The priority of an event does not often need to be set or referenced.

veventWindowPointerDragEvent

        typedef ... veventWindowPointerDragEvent;
A subclass of veventWindowPointerEvent indicating that the pointer has moved. Not all pointer motion will create events. Pointer motion while a button is depressed will usually create drag and detail events. Pointer motion when no button is depressed will create motion events only if the window has motion events enabled. See also vwindowSetMotion.

Generally, the window associated with the event will be the window the pointer is over at the time of the event, although it could be some other window if input is grabbed. See also vwindowGrabInput Instances of veventWindowPointerDragEvent may be either drag, detail or motion events. The event's type (Accessed with vevent::GetType and vevent::SetType) will be either veventPOINTER_DRAG, veventPOINTER_DETAIL, or veventPOINTER_MOTION.

Drag and motion events always reflect the current position of the pointer. Detail events reflect every place the pointer has been. If your application needs to know every place the pointer has been (for instance, in a paint program,) it should handle detail events. If your application is only concerned with the current position of the pointer (for instance, when dragging an object), it should handle drag and/or motion events. See also vwindowHANDLE_DETAIL, vwindowHANDLE_DRAG, and vwindowHANDLE_MOTION.

veventCancelQuit

        void veventCancelQuit();
When responding to a command issued in the scope of the application and bound to vname_Quit, the application should do only one of two things. When the logical shutdown sequence is finished (when all open files have been saved or discarded, or the sequence has been cancelled) either one or the other of two procedures should be called. Call veventStopProcessing if the application is ready to quit, or call veventCancelQuit if the application is not yet ready to quit. (Call one or the other of these procedures after the logical shutdown sequence has been completed or cancelled, not necessarily just when a value is returned to the command issuer.)

For example: 


switch(selector) { 
case commandQUIT: 
    if (closedAllFiles())
          veventStopProcessing();
    else
          veventCancelQuit(); 
    return TRUE;
}

The procedure handling the issue of the command should always return vTRUE, indicating that the command was understood. This procedure is useful in implementing correct session management under the various window managers. See also vobjectIssueCommand, vapplicationSetSpace, and vcommandSpace.

veventInitFDIterator

        void veventInitFDIterator(
                veventFDIterator * iterator);
Initializes the passed in veventFDIterator for use in iterating over all currently existing veventFDs. The veventFDIterator that is passed in must be a pointer to enough memory to hold a veventFDIterator. This is usually a local variable of type veventFDIterator. Once initialized, the iterator can be incremented with veventNextFDIterator. Each iteration's veventFD can be had with veventGetFDIteratorFD, and once the iteration is finished, the iterator can be destroyed with veventDestroyFDIterator.

void sample( void )
{
    veventFDIterator         iterator;
    veventFD                *pFD;

    veventInitFDIterator( &iterator );

    while (veventNextFDIterator( &iterator ))
    {
        pFD = veventGetFDIteratorFD( &iterator );

        /* lots of code here */
    }

    veventDestroyFDIterator( &iterator );
}

veventStopProcessing

        void veventStopProcessing();
Marks the currently running event loop as done. After dispatching all currently queued events, veventProcess returns control to the procedure that started it (with veventProcess).

veventUnpost

        vbool veventUnpost(
                vevent * event);
Attempts to remove the event from the current thread's event queue. Returns vTRUE if it succeeded.

Opposite of veventPost and its many deriviatives. This does not destroy the event, nor is a veventUnreference performed. Ownership of the event is presumed to have passed from the event queue to the caller.

veventUnreference

        void veventUnreference(
                vevent * event);
Releases a reference to an event. If the event has no more references to it, it may be destroyed. Opposite of veventReference.

veventUnregisterFD

        void veventUnregisterFD(
                veventFD * file);
Unregisters the given veventFD from the event system. The passed in veventFD is implicitly destroyed by veventUnregisterFD and cannot be used after this call is made. Any notify procedures set on the given veventFD will be dismissed. See also veventRegisterFD.

veventUnRegisterFD only works on UNIX platforms.

veventValidate

        vbool veventValidate(
                const vevent * event);
Returns vTRUE if the event is a valid event. If the event is not valid it may return vFALSE or it may not return at all. The process might even crash if the pointer is bad. See also veventAssert and veventHandledAssert.
