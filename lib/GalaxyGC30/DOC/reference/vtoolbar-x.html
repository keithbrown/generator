vtoolbar

        typedef ... vtoolbar;
Contains the information needed to represent a single toolbar; objects of this type have classes derived from vtoolbarClass.

Contains the information needed to represent a single toolbar. A toolbar is a kind of container designed to hold simple dialog items which present core application functionality to the user in a convenient manner. Toolbars are designed to work well inside of a docker as well as inside a normal item list. A toolbar does not use a spring solution to lay out its items, rather it lines the items up in rows putting as many items as will fit on each row. Row breaks only occur where a vtoolbarSeparator is present. Toolbars can dynamically switch between horizontal and vertical layouts. Refer to the section SetVertical for more information on this topic.

Additionaly, every toolbar has the potential to be user-rearrangeable. This means that the application user is able to rearrange the items in the toolbar's item list with drag and drop operations. The capacity for the user to add, remove, and move items around within a toolbar and between toolbars is built into the toolbar class. Refer to the sections SetRearrangeable and vtoolbarRearranger for more information.

vtoolbarRearrangeOccurrence

        typedef ... vtoolbarRearrangeOccurrence;
Contains the information needed to represent a single toolbar rearrange occurrence. Objects of this type have classes derived from vtoolbarRearrangeOccurrenceClass. Toolbar rearrange occurrences are announced by a rearranger through the toolbar they serve. All the information relevant to a rearrange operation can be obtained from a rearrange occurrence. Refer to the section vtoolbarGetRearrangerOwner for more information.

Toolbar rearrange occurrences can be observed by installing an observer on the toolbar with vobservableInstallObserver or by overriding the toolbar's Observe method. These announcements are provided to facilitate certain simple customizations of the rearrange behavior. For instance, you can watch for occurrences of type vtoolbarREARRANGE_PRE_REMOVE_OCCURRENCE to get a pointer to an item that is going to be removed, copy it, and create a new floating toolbar that contains just that item. Another simple way to customize rearrange behavior is with the vtoolbarSetRearrangerPermissions method. Refer to the sections GetType and GetDragItem for more information.

vtoolbarRearranger

        typedef ... vtoolbarRearranger;
Contains the information needed to represent a single toolbar rearranger object. Objects of this type have classes derived from vtoolbarRearrangerClass, which is in turn derived from vobservableClass.

A toolbar rearranger is a non-graphical object which encapsulates the logic necessary to allow end users to rearrange toolbars. The three most important methods of a rearranger which perform the core work of rearranging are:

  • ProcessRearrange
  • TestRearrange
  • Rearrange.
Several other important methods listed below are available to customize the behavior of a rearranger.
  • SetPermissions
  • SetBindingMap
  • SetBindingMapOwned.
In terms of object interactions, a toolbar rearranger serves as an intermediary between a rearrange dropsite and a toolbar. It synchronizes the dropsite's rectangle with the toolbar's item rectangle and manipulates the toolbar's item list when a rearrange drag lands on a rearrange dropsite.

vtoolbarSeparator

        typedef ... vtoolbarSeparator;
Contains the information needed to represent a single toolbar separator. Objects of this type have classes derived from vtoolbarSeparatorClass. A toolbar separator is simple dialog item which is used to make space between groups of items in a toolbar's item list. When a toolbar lays out its items, it uses the separator's GetHeight and GetWidth methods to determine how large the separation should be. By default, separators are invisible, but subclasses can draw them however they wish. Row breaks in a toolbar can only occur on toolbar separators.

vtoolbarDetermineInsertionIndex

        int vtoolbarDetermineInsertionIndex(
                vtoolbar * targetItem,
                const vrect * dragRect,
                unsigned short * placementHint);
Returns the best item list index for inserting a new item into the toolbar based on the rectangle parameter. This method is used by the toolbar's rearranger to determine how to place the items in a toolbar's item list after a rearrange drag has been dropped. The rectangle typically represents the drag outline of the dialog item being rearranged. For correct behavior it should be in coordinates of the toolbar's item list.

The value returned is the best index for insertion or -1 if no insertion should take place. Additionally, if placementHint is non-NULL its value will be set to a "hint" about the placement of the new item. Possible values returned are defined below.

vtoolbarINSERTION_HINT_NONE
The default value. Indicates that no special handling is necessary in rearranging the item list.
vtoolbarINSERTION_HINT_ROW_END
Indicates that the rectangle is located at the end of a row of toolbar items. Since rows are delimited by [[toolbar separators]], every row except for the last ends in a separator. If an item were to be inserted after a row-terminating separator, then the item would appear at the beginning of the next row rather than the end of the expected row.
vtoolbarINSERTION_HINT_PAST_ROW_END
Implies a situation similar to vtoolbarINSERTION_HINT_ROW_END except the drag rectangle is significantly past the end of a row. When toolbar rearrangers see this hint, they insert an extra vtoolbarSeparator before the dropped item to provide some separation between the dropped item and the end of its new row.

vtoolbarDetermineRearrangerBinding

        void vtoolbarDetermineRearrangerBinding();
Returns the rearrange binding associated with a particular set of event modifiers. The return value is a globally interned vname equal to one of the following:
  • vtoolbar_RearrangeCopy (or vnameInternGlobalLiteral("RearrangeCopy")
  • vtoolbar_RearrangeMove (or vnameInternGlobalLiteral("RearrangeMove")
  • vtoolbar_RearrangeNone (or vnameInternGlobalLiteral("RearrangeNone")
The default implementation of this method looks up the specified modifiers in the rearranger's binding map as determined by DetermineBindingMap (see also BindingMap and GetBindingMap). If there is a binding in the binding map corresponding to the specified modifiers, that binding is returned, otherwise the value stored under the key vtoolbarREARRANGE_BINDING_MAP_DEFAULT_KEY is checked. It is returned if non-NULL. If no binding is found, the vtoolbar_RearrangeNone binding is returned.

The rearranger uses this method whenever it needs to determine the binding for some particular event modifiers. If the vdict lookup semantics are flexible enough for the desired result, there is no need to override this method. On the other hand, if full control is needed, then overriding this method can be useful. For instance, an override of this method could essentially turn off all toolbar rearranging temporarily based on the state of the application by returning vtoolbar_RearrangeNone when rearranging is off.

This method is implemented by sending the message vtoolbarRearrangerDETERMINE_BINDING.

  • The binding for the event modifiers as determined by a call to DetermineBinding is neither "RearrangeCopy" nor "RearrangeMove".
  • The x and y of the event, (refer to veventPointerX and veventPointerY), are not over an item in the owner item's item list.(Refer to GetOwner).
If all of the tests above are passed, then the method creates a rearrange drag using CreateDrag, sets the type to vtoolbarREARRANGE_DRAG and calls TestRearrange to see if the operation is allowed. If TestRearrange returns FALSE then the process will be aborted and this method will return FALSE. Otherwise, the rearranger will announce the start of the rearrange with a vtoolbarRearrangeOccurrence of type vtoolbarREARRANGE_START_OCCURRENCE and then process the rearrange drag by calling vdragProcessEngineDrag.

If the call to vdragProcessEngineDrag returns FALSE (meaning that the rearrange drag was dropped over an invalid dropsite, then the rearranger attempts to remove the drag item from its itemlist. The removal proceeds as follows:

  • The type of the drag is set to vtoolbarREARRANGE_REMOVE,
  • TestRearrange is called to see if the operation is allowed, and if so,
  • Rearrange is called to perform the action.

This method is implemented by sending the message vtoolbarRearrangerPROCESS_REARRANGE to the rearranger.

A toolbar rearrange is a transaction that involves several parties, including a container item like a toolbar, a rearranger, and a dropsite; all on both source and target ends of the rearrange. Additionally there is the dialog item from the source container which is being rearranged.

The rearrange drag object is the glue that binds these disparate elements together for the duration of the rearrange action, but rather than keep a pointer to both source and target containers, source and target dropsites, and source and target rearrangers, the drag just keeps track of the source rearranger and target rearranger, since pointers to the other objects can be obtained indirectly through them.

Application writers typically do not need to call this function directly; the target rearranger is automatically set by the rearrange dropsite over which the drag is dropped.

The function vtoolbarSetRearrangeDragTargetRearranger is implemented by sending this message to the rearrange drag.

vtoolbarRearrangerBINDING_MAP

        <vtoolbarRearrangerBINDING_MAP>
Contains the binding map for a toolbar rearranger class or NULL if the class has no binding map. The binding map is a dictionary of type vdict which maps a keys of type vkeyModifiers to binding values of type vname. The binding indicates what rearrange operation should be performed as a result of an event with the specified modifiers. The bindings understood by toolbars and toolbar rearrangers are ``RearrangeCopy'', ``RearrangeMove'', and ``RearrangeNone''.

The function vtoolbarDetermineRearrangerBindingMap first checks the rearranger object for an instance-level binding map, and if not found, returns this class-level binding map.

The rearranger itself only refers to the binding map in the method DetermineBinding which is the proper way to determine the rearrange binding for a particular set of event modifiers. The rearrange binding map should only be used directly when storing or removing bindings.

        <vtoolbarRearrangerCREATE_TOOLBAR_SEPARATOR>
Creates a toolbar separator of the appropriate class for this rearranger's toolbar. A toolbar rearranger ocassionaly must create a separator item to insert into its toolbar item list as it rearranges the toolbar's items. This is, for example, the action taken by the rearranger when a user drags an item just slightly to the side. Inserting a separator gives the visual impression that the item was actually moved in the direction that the user dragged.

Rearranger subclasses wishing to use a customized toolbar separator subclass can override this method. The default implementation of this method returns a newly initialized vtoolbarSeparator object of the default class.

The function vtoolbarCreateRearrangerToolbarSeparator is implemented by sending this message to the rearranger.

vtoolbarRearrangerDETERMINE_BINDING

        <vtoolbarRearrangerDETERMINE_BINDING>
Returns the rearrange binding associated with a particular set of event modifiers. The return value is a globally interned vname equal to one of the following:
  • vtoolbar_RearrangeCopy (or vnameInternGlobalLiteral("RearrangeCopy")
  • vtoolbar_RearrangeMove (or vnameInternGlobalLiteral("RearrangeMove")
  • vtoolbar_RearrangeNone (or vnameInternGlobalLiteral("RearrangeNone")
The default implementation of this method looks up the specified modifiers in the rearranger's binding map as determined by DetermineBindingMap (see also BindingMap and GetBindingMap). If there is a binding in the binding map corresponding to the specified modifiers, that binding is returned, otherwise the value stored under the key vtoolbarREARRANGE_BINDING_MAP_DEFAULT_KEY is checked. It is returned if non-NULL. If no binding is found, the vtoolbar_RearrangeNone binding is returned.

The rearranger uses this method whenever it needs to determine the binding for some particular event modifiers. If the vdict lookup semantics are flexible enough for the desired result, there is no need to override this method. On the other hand, if full control is needed, then overriding this method can be useful. For instance, an override of this method could essentially turn off all toolbar rearranging temporarily based on the state of the application by returning vtoolbar_RearrangeNone when rearranging is off.

The method vtoolbarDetermineRearrangerBinding is implemented by sending this message to the rearranger.

vtoolbarRearrangerINSERT_ITEM

        <vtoolbarRearrangerINSERT_ITEM>
Inserts dropItem into dropTarget's item list based on the rectangle. The rectangle should be in terms of dropTarget's coordinate system. The default implementation of this method only works if dropTarget is a vtoolbar, or a subclass thereof.

This method is called by the rearranger to insert dialog items into the item list of its toolbar in response to user rearrange drag and drop actions. Part of the work of determining how to insert the item into a toolbar's item list is done by calling vtoolbarDetermineInsertionIndex.

The method vtoolbarRearrangerInsertItem is implemented by sending this message to the rearranger.

vtoolbarRearrangerMOVE_ITEM

        <vtoolbarRearrangerMOVE_ITEM>
Moves dropItem from one place in dropTarget's item list to another based on the rectangle. The rectangle should be in terms of dropTarget's coordinate system. The default implementation of this method only works if dropTarget is a vtoolbar, or a subclass thereof.

This method is called by the rearranger to move dialog items within the item list of its toolbar in response to user rearrange drag and drop actions. Part of the work of determining how to move the item is done by calling vtoolbarDetermineInsertionIndex.

The method vtoolbarRearrangerMoveItem is implemented by sending this message to the rearranger.

vtoolbarRearrangerPROCESS_REARRANGE

        <vtoolbarRearrangerPROCESS_REARRANGE>
Processes a rearrange action initiated by the specified event. The event passed in is typically a pointer button down event. The return value indicates the success of the operation, TRUE if successful, FALSE if not. The return value will be FALSE and no action will be taken if any of the following is true:
  • The binding for the event modifiers as determined by a call to DetermineBinding is neither "RearrangeCopy" nor "RearrangeMove".
  • The x and y of the event, (refer to veventPointerX and veventPointerY), are not over an item in the owner item's item list (see GetOwner).
If all of the tests above are passed, then the method creates a rearrange drag using CreateDrag, sets the type to vtoolbarREARRANGE_DRAG and calls TestRearrange to see if the operation is allowed. If TestRearrange returns FALSE then the process will be aborted and this method will return FALSE. Otherwise, the rearranger will announce the start of the rearrange with a vtoolbarRearrangeOccurrence of type vtoolbarREARRANGE_START_OCCURRENCE and then process the rearrange drag by calling vdragProcessEngineDrag. If the call to vdragProcessEngineDrag returns FALSE (meaning that the rearrange drag was dropped over an invalid dropsite), then the rearranger attempts to remove the drag item from its itemlist. The removal proceeds as follows:
  • The type of the drag is set to vtoolbarREARRANGE_REMOVE
  • TestRearrange is called to see if the operation is allowed, and if so,
  • Rearrange is called to perform the action.

    The function vtoolbarProcessRearrangerRearrange is implemented by sending this message to the rearranger.

vtoolbarRearrangerREARRANGE

        <vtoolbarRearrangerREARRANGE>
Rearranges the rearranger's owner item list based on the rearrange drag and the coordinates specified by the event. The event is typically the pointer button up event which ends a rearrange drag.

Most of the parameters of the operation are obtained from the attributes of the rearrange drag. The method acts on its owner item's item list depending upon the type of the rearrange drag. Refer to the section vtoolbarGetRearrangeDragType for more information.

Code which calls this method should in general first call TestRearrange to see if the rearranger wants to handle the given rearrange action. Code like the following example will suffice. 

int callRearrange( vtoolbarRearranger *rearrange, 
                   vtoolbarRearrangeDrag *drag,
                   vevent *event )
{
     if ( vtoolbarRearrangerTestRearrange( rearrange, drag, event ) )
     {
          vtoolbarRearrangerRearrange( rearrange, drag, event );
          return TRUE;
     }
     return FALSE;
}
The default implementation of this method announces vtoolbarRearrangeOccurrences both before and after modifying the owner's item list. These can be observed by installing an observer on the owner, not the rearranger itself. Refer to vobservable).

The function vtoolbarRearrangerRearrange is implemented by sending this message to the rearranger.

vtoolbarRearrangerSET_DROPSITE

        <vtoolbarRearrangerSET_DROPSITE>
Sets the dropsite to be used by the rearranger. The rearranger ensures that this dropsite's rectangle mirrors the rectangle of the rearranger's owner item. It is generally not necessary to set a dropsite since a rearranger will automatically create a dropsite using CreateDropsite if one is not provided. When a rearrange drag is dropped over the rearranger's dropsite, the dropsite will invoke the rearranger's (and possibly one other rearranger's) TestRearrange and Rearrange methods to perform the rearrange action.

The dropsite is owned by the rearranger and will be destroyed automatically when the rearranger is destroyed, or when a different dropsite is set on the rearranger.

The function vtoolbarSetRearrangerDropsite is implemented by sending this message to the toolbar.

vtoolbarRearrangerSET_OWNER

        <vtoolbarRearrangerSET_OWNER>
Sets the item which owns this rearranger. The owner of a rearranger is the the item for which the rearranger provides rearrange services. Ideally the owner of a rearranger could be any dialog item, but the current implementation limits it to being either a vtoolbar or vtoolbar subclass.

The owner of the rearranger is responsible for destroying it when it is no longer needed.

The function vtoolbarSetRearrangerOwner is implemented by sending this message to the toolbar.

vtoolbarRearrangerSET_PERMISSIONS

        <vtoolbarRearrangerSET_PERMISSIONS>
Sets the rearrange actions that the rearranger will allow. The rearranger permissions are designed to provide a simple yet fairly flexible way to customize the actions taken by a rearranger. The default implementations of TestRearrange and Rearrange use these permissions to determine what action to take. For greater control, one can subclass the default rearranger class and override the TestRearrange and Rearrange methods. Alternatively, depending on what sort of customization is desired, one can install an observer on the rearranger's owner item and watch for occurrences of type vtoolbarRearrangeOccurrence.

The method vtoolbarSetRearrangerPermissions is implemented by sending this message to the toolbar. Refer to the section vtoolbarSetRearrangerPermissions for more information.

vtoolbarRearrangerTEST_REARRANGE

        <vtoolbarRearrangerTEST_REARRANGE>
Queries to see if the rearranger will allow the specified action. Returns TRUE if the action is allowed and FALSE if not. The action is specified primarily by the action type of the vtoolbarRearrangeDrag parameter. Refer to vtoolbarRearrangeDrag for more information. The event is not used by the default implementation, but subclasses which override this method can use attributes of the event such as the x and y coordinates and active key modifiers.

The default implementation of this method returns TRUE or FALSE based on the type of the drag and the rearranger's permission flags. Refer to SetPermissions for more information. The rearranger's Rearrange method will not be called if this method returns FALSE.

The function vtoolbarRearrangerTestRearrange is implemented by sending this message to the rearranger.

vtoolbarSET_REARRANGEABLE

        <vtoolbarSET_REARRANGEABLE>
If rearrange is TRUE, then end users will be able to rearrange the toolbar using drag and drop, if FALSE this functionality will be disabled.

If the rearrange functionality is being enabled, but the toolbar does not have a rearranger (set via SetRearranger, then a rearranger of the default class will automatically be created by calling CreateToolbarRearranger.

Any rearranger set on the toolbar will not be removed or destroyed when rearrange capability is disabled or re-enabled. Because a toolbar owns its rearranger, the rearranger will be automatically destroyed when the toolbar is destroyed or when a new rearranger is set.

The function vtoolbarSetRearrangeable is implemented by sending this message to the toolbar.

vtoolbarSET_VERTICAL

        <vtoolbarSET_VERTICAL>
Changes the orientation of the toolbar to vertical if vertical is TRUE or horizontal if it is FALSE. A vertical toolbar lays its items out in columns rather than rows. Toolbars in dockers automatically change their orientation to match the orientation of their docker.

The function vtoolbarSetVertical is implemented by sending this message to the toolbar.

vtoolbarSET_WRAP

        <vtoolbarSET_WRAP>
Enables wrapping if wrap is TRUE or disables wrapping if wrap is FALSE. Wrapping means that items on the end of a row of items in the toolbar will automatically move down to the next row to avoid being clipped by the edge of the toolbar, much like in a word processor where words at the end of a line wrap to next line if they do not fit. Wrapping breaks occur only at items of type vtoolbarSeparator.

The behavior of a wrapping toolbar depends on its docked status. A wrapping toolbar that is not in a docker will wrap items to fit within the toolbar's item rectangle. A wrapping toolbar in a docked docker will occupy the entire span of the dockitem, and the items in the toolbar will wrap instead of clip. The behavior of a toolbar in a floating docker is not affected.

The function vtoolbarSetWrap is implemented by sending this message to the toolbar.
