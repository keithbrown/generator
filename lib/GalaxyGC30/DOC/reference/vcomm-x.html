An exception type generated by vcommOpen and vcommListen when there is an error with a vcommAddress specification. vcommBase

Abstract base class for transport objects. The vcommBase class specifies functionality for generic interprocess communication. Subclasses of vcommBase implement different transport mechanisms for interprocess communication. Only the functionality shared by both connection-oriented and message-based transports is specified in the vcommBase class.

vcommBaseException

        typedef ... vcommBaseException;
Exceptions of class vcommBaseException are thrown when errors occur when invoking the following methods:

  • vcommEnableBase
  • vcommDisableBase
  • vcommReceiveBase
  • vcommSendBase
  • vcommHaltBase
  • vcommWaitBase
  • vcommCopyBase
From the exeption, a general error code can be obtained which describes the nature of the error. These error codes include:

vcommERROR_HALT
A blocking operation was interrupted by Halt.
vcommERROR_ADDRINUSE
The address location that this transport object is being enabled on is being used by another tranport object.
vcommERROR_CONNECTION
The peer no longer exists or has disconnected.
vcommERROR_STATE
The operation can not be performed in the current state of the transport object; for example, attempting to call Send in the vcommDISABLED state.
vcommERROR_TRANSPORT
A system call has failed. Obtain the vexERRNO attribute to the exception to obtain the system specific error code.

vcommBaseNoteProc

        typedef ... vcommBaseNoteProc;
Denotes a vcommBase Notify function that is called when significant events occur. A typical notification method looks like the following: 

void myNoteProc(vcommBase *base, vcommEvent *event)
{
  switch (vcommGetEventType(event)) 
  {
   case vcommENABLED:
    /* ready to send and receive data */
    break;

   case vcommRECEIVE:
    /* data is available to receive */
    break;

   case vcommSEND:
    /* data can be sent */
    break;

   case vcommDISABLED:
    /* disabled from doing communication */
    break;

   case vcommERROR:
     /* something has gone wrong */
     switch (vcommGetEventError(event))
     {
      case vcommERROR_NONE:
       /* unknown error */
       break;
      case vcommERROR_TRANSPORT:
       /* a system error has occurred */
       break;
      case vcommERROR_CONNECTION:
       /* the connection to peer has been lost (streams only) */
       break;
     }
   break;
   }
  }

vcommAddInputQueue

        void vcommAddInputQueue(
                vcommChannel * channel,
                vbyte * buffer,
                size_t nBytes);
Queues an input request for nBytes bytes into buffer. This is called by the driver function implementing Read when all the requested data has not arrived yet. nBytes is the number of bytes remaining, and buffer is the buffer passed to the Read function plus the number of bytes read into it so far.

This routine should be used only by transport drivers. It is not needed in normal application code. Refer to ``Writing a Transport Driver'' in the ``Communication Manager'' chapter of the Galaxy Application Environment Programmer's Guide for an example of how this routine may be used.

vcommAdvanceVector

        void vcommAdvanceVector(
                vcommVector * source,
                vcommVector * dest,
                int less_bytes,
                int * nvectors);
Convenience method which indexes an array of vectors source with a second array of vectors dest minus less_bytes of data. nvector as an in-parameter indicates the number of vectors of source points to. nvector as an out-parameter indicates the number of vectors dest points to that actually point to data.

For example, source is an array of three vectors, each of which contain a 100 byte buffer of data. less_bytes is set to 150. dest is an array of three vectors. nvectors is set to 3. After the invocation of vcommVector, source is unchanged and dest still points to 3 vectors itself. However, vector 1 of dest points to the last half of vector 2 of source and vector 2 of dest is equivalent to vector 3 of source. In addition, nvectors is 2, indicating the number of vectors of dest that actually point to data.

Note that both source and dest must point to at least nvectors vectors when the method is invoked. Also, the total number of bytes of data pointed to by source must be greater than or equal to less_bytes.

vcommAvail

        size_t vcommAvail(
                vcommChannel * channel);
Returns the number of bytes waiting to be read on the physical channel. This is independent from the number of bytes which are queued to be read. Actual message boundaries are ignored and complete messages may not have arrived yet. To know the actual size of a message requires a higher level protocol imposed on the channel. For an example, refer to the examples in ``Presentation'' in ``Using the Communication Manager'' chapter in the Galaxy Application Environment Programmer's Guide. This function calls the Avail function for the channel's transport driver.

Exceptions: vcommNotOpenException, vcommChannelException

vcommDisableBase

        void vcommDisableBase(
                vcommBase * base);
Disables an enabled vcommBase object from communication. Refer to the section the sections vcommDisableStream and vcommDisableDatagram for further information regarding the use of Disable with vcommStream and vcommDatagram objects.

If the disabling operation completes immediately, a vcommDISABLED message is sent to the notify method. If however, the disabling operation cannot complete immediately, one of the following occurs:

If set to asynchronous:
The vcommBase object enters the vcommDISABLING state to later notify the observe method with a vcommDISABLED message when complete.
If set to synchronous:
Execution of the thread is blocked until the disabling operation is able to complete. When complete, the vcommBase object is in the vcommDISABLED state.
If an error occurs, an exception of type vcommBaseException is thrown with one of the following error codes:
  • vcommERROR_TRANSPORT
  • vcommERROR_STATE

Refer to the section vcommBaseException and the definition of each error code for further information.

This function is implemented by sending the message vcommBaseDISABLE to the vcommBase object.

Requests that the currently disabled vcommBase object be enabled for communication. Refer to the section the section vcommEnableStream and vcommEnableDatagram for further information regarding the use of Enable with vcommStream and vcommDatagram objects.

If the enabling operation is able to complete immediately, a vcommENABLED message is sent to the notify method. If however, the enabling operation cannot complete immediately, one of following situations occurs:

If set to asynchronous:
The vcommBase object enters the vcommENABLING state to later notify the observe method with a vcommENABLED message when complete.
If set to synchronous:
Execution of the thread is blocked until the enabling operation is able to complete. When complete, the vcommBase object is in the vcommENABLED state.
If an error occurs, an exception of type vcommBaseException is thrown with one of the following error codes:
  • vcommERROR_TRANSPORT
  • vcommERROR_STATE
  • vcommERROR_ADDRINUSE

Refer to the section the section vcommBaseException and the definition of each error code for further information.

This function is implemented by sending the message vcommBaseENABLE to the vcommBase object.

vcommEnableDatagram

        void vcommEnableDatagram(
                vcommDatagram * dgram);
Invoking Enable on a vcommDatagram object enables the object to do communication at the location specified by the local locator (referenced by vcommGetDatagramLocator).

Once enabled, the vcommDatagram object becomes receptive to data messages sent to the location specified by its local locator. In addition, a data message sent from the object contains this locator information, allowing the receptive datagram transport object(s) the ability to identify the sender of the data message.

Refer to the section vcommEnableBase for general behavior of Enable on both stream and datagram transport objects. This function is implemented by sending the message vcommBaseENABLE to the vcommDatagram object.

vcommEnableStream

        void vcommEnableStream(
                vcommStream * stream);
Enables a vcommStream object for communication by establishing a connection with another stream transport object. Establishing a connection is acheived in one of two ways:
  • Listen for a connection.
  • Connect to a listener.

To listen for a connection, make certain that the remote locator of the vcommStream object (referenced by vcommGetStreamRemoteLocator) is unspecified. When Enable is invoked, the vcommStream object will listen for a connection at the location specified by the local locator (referenced by vcommGetStreamLocator). If the local locator is also unspecified when Enable is invoked, a location is automatically picked and can be examined upon completion of Enable. The remote locator identifies the location of the peer after the connection has been established: When the vcommStream object enters the vcommEnabled state.

To connect to a listening stream transport object, specify the address of the peer with the remote locator. When Enable is invoked, the vcommStream object is bound to the location specified by the local locator. This location is automatically picked if not specified. A connection is then attempted with the object listening at the location specified by the remote locator.

vcommClearLocator may be used to clear the attributes of a locator so that it does not specify an address. Refer to the section vcommEnableBase for general behavior of Enable on both stream and datagram transport objects. This function is implemented by sending the message vcommBaseENABLE to the vcommStream object.

vcommFeedInputQueue

        size_t vcommFeedInputQueue(
                vcommChannel * channel,
                vbyte * buffer,
                size_t nBytes);
Gives the Communication Manager new input to satisfy reads queued by vcommAddInputQueue. nBytes is the number of new bytes available in buffer. buffer points to the first newly available byte. Any bytes consumed by the channel queue will by copied out of buffer and into the buffer which was passed to vcommAddInputQueue.

The function returns either when no more input is left queued, or all the new bytes have been consumed. The result is the number of bytes that were actually processed. The first byte available for the driver to use is (buffer + result), and there are (nBytes - result) bytes following. As each queued read is completed, a vcommREAD_COMPLETE message is sent to the channel's notification function.

This routine should be used by transport drivers, however is not needed in normal application code. Refer to ``Writing a Transport Driver'' in the ``Communication Manager'' chapter of the Galaxy Application Environment Programmer's Guide for an example of how this routine may be used.

vcommInitAddress

        void vcommInitAddress(
                vcommAddress * address);
Initializes an address that is allocated on the program stack, not the one created in the dynamic heap as the result of vcommCreateAddress. This call must occur before the address can be used in any other address function, and should be destroyed with vcommDestroyAddress when it is no longer needed to avoid memory leaks in internals of the address structure.

The default values of the location and port are NULL, and the default transport is vnameNULL. Note that vnameNULL is an invalid transport so vcommSetAddressTransport must eventually be called. If the channel is to be opened, the particular port to connect to must be set by calling vcommSetAddressPort, and if the location to connect to is not local, it must be set with vcommSetAddressLocation. The address passed to vcommListen does not need to modify the location, nor does it need to modify the port if it does not care which particular port is used.

Example usage: 


{
	vcommAddress	address;
	vcommInitAddress(&address);
	...
	/*
	 * Assume address was used above
	 * and is no longer needed.
	 */
	vcommmDestroyAddress(&address);
} /* end code fragment */

vcommReceiveDatagram

        size_t vcommReceiveDatagram(
                vcommDatagram * dgram,
                vcommLocator * locator,
                vcommVector * vector,
                int num);
Receives a data message sent to the transport object and puts the data into the vectors specified by the array vector. The number of vectors is indicated by num. If successful, the return value indicates the total number of bytes received and locator specifies the address of the sender.

If the transport object is set to synchronous, execution of the thread is blocked until a data message can be received. Asynchronous operation attempts to receive a data message, but returns immediately regardless. If a message was not received, a vcommRECEIVE message is sent to the transport object when a data message become available.

If the buffer space provided is not large enough, the data message is truncated. Depending on the platform and the specific transport, the remaining data may be discarded or available the next time Receive or ReceiveBuffer is called.   • vcommERROR_TRANSPORT

vcommEXCEPTION

        vcommEXCEPTION
Sent to a channel's notification function when a channel encounters an exception during an asynchronous read or an asynchronous write. This does not include an exception generated from the notification function during the vcommREAD_COMPLETE or vcommWRITE_COMPLETE messages. The exception is stored in the current exception chain and is otherwise not any different from other uses of the Exception Manager.

The exception should not be propagated unless the intention is to exit veventProcess with the exception. This is because asynchronous reads come from a thread out of the event loop and do not necessarily have the original vcommRead/vcommWrite context to catch the exception. Only the oldest queued read or write should be affected by each exception.

vcommINPUT

        vcommINPUT
Sent to a channel's notification function when a connection has received new data to be read. It is up to the notification function to call vcommRead vcommChannel to consume the data queued on the channel. If it does not, then the data will not be lost; however, the notification function will not be called again until more data (in addition to what has not been read yet) is received.

vcommAvail can be called to query the channel for how much input is waiting to be read. The result can be bigger than the actual message if there are multiple messages waiting to be read, or smaller than the actual message if the whole message has not arrived yet. For more information about this, refer to ``Notification Functions'' in the ``Communication Manager'' chapter of the Galaxy Application Environment Programmer's Guide.

If there are reads queued on a channel, vcommINPUT will not be issued until all the queued reads complete (and issuesad section ``Using the Communication Manager'' in the vcommREAD_COMPLETE) and there is additional data waiting to be read.

vcommOPTION_SYNCHRONOUS

        vcommOPTION_SYNCHRONOUS

        vcommOPTION_INPUT_BUFFER_SIZE

        vcommOPTION_OUTPUT_BUFFER_SIZE

        vcommOPTION_REUSEADDR
The synchronous option is used to make channel I/O synchronous (or asynchronous). When a channel is synchronous, vcommRead and vcommWrite will block until they complete. When a channel is asynchronous, vcommRead and vcommWrite will queue the uncompleted I/O and finish completing while the application is otherwise idle in the event loop. Drivers are asynchronous by default (option value is FALSE). Note that asynchronous semantics of the Read/Write return results still hold when the channel is synchronous, however the value returned will always be the value requested.
