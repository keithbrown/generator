Types

vservicechsr

        typedef ... vservicechsr;
A service chooser is a DAS component that displays a list of all services of which the local broker is aware. A configurable list of service attributes is also displayed for information purposes.

vservicechsrApplyFunc

        typedef ... vservicechsrApplyFunc;
Denotes an apply function that is called when a service has been chosen from the service chooser serviceChooser. scrap is a dictionary scrap that contains the attribute name and value pairs for each named attribute of the service. The apply function returns TRUE if the service was accepted and the chooser can close its dialog, or FALSE if the service was rejected and the chooser dialog should remain open.

vservicechsrClass

        typedef ... vservicechsrClass;
Contains method and variable bindings for objects of type vservicechsr. The service chooser class controls service display and filtering by attribute. Classes declared with this type contain the same set of methods and variables as the default service chooser class, but the behavior of methods and the contents of variables may vary. This class is derived from vconfirmClass.

vservicechsrClose

Closes the service chooser serviceChooser so it is no longer visible. This function has no effect on a service chooser that is already closed. This function is implemented by sending the message vwindowCLOSE to serviceChooser.

vservicechsrGetMatchScrap

        vscrap * vservicechsrGetMatchScrap(
                vservicechsr * serviceChooser);
Returns the dictionary scrap sent to the local service broker during match requests by the service chooser serviceChooser. The default is a dictionary scrap with no elements in it; this matches every service currently registered with the broker. Ownership of the scrap is retained by the chooser; the scrap should not be destroyed by the application. To change the attributes to be matched against, use vservicechsrSetMatchScrap.

vservicechsrRescan

        void vservicechsrRescan(
                vservicechsr * serviceChooser);
Queries the local service broker for a new list of registered services. This function is implemented by sending the message vservicechsrRESCAN to serviceChooser. The query sends a statement to the local service broker and displays the results of the query when they are sent back to the application.

Note that a rescan operation may take a noticeable amount of time, due to network loading and routing delays.

vservicechsrSetApply

        void vservicechsrSetApply(
                vservicechsr * serviceChooser,
                vservicechsrApplyFunc applyFunc);
Sets the apply function applyFunc on the service chooser serviceChooser. This function is implemented by sending the message vservicechsrSET_APPLY to serviceChooser. The apply function is called by vservicechsrCallApply when the service chooser's Apply button is pressed.

See the vservicechsrCallApplyFunc section for more information.

vservicechsrSetMatchScrap

        void vservicechsrSetMatchScrap(
                vservicechsr * serviceChooser,
                vscrap * scrap);
Sets the dictionary scrap sent to the local broker during match requests by the service chooser serviceChooser to the scrap scrap. The default matching scrap is an empty dictionary scrap; this matches all services registered with the broker. Ownership of the scrap is passed to the chooser by this function. The scrap should be a dictionary scrap with zero or more elements in it. The elements are sent to the local broker at the next rescan operation, and only those services that match the specified attributes are returned by the broker to the chooser for display.

A service is said to match a set of attributes when the service has each of the attributes specified in the scrap; however, the service may have additional attributes that were not part of the match request.

vservicechsrStartup

        void vservicechsrStartup();
This function explicitly initializes the entire Service Chooser class. Most applications need not call this function, as the Service Chooser class is self-initializing. Only those applications with strict concerns about module startup ordering and resource usage need be concerned with this function.

vservicechsrNEXT_SELECT

        vservicechsrNEXT_SELECT
This constant is provided so that selector values defined by subclasses do not conflict with those defined by their superclasses. Only class-level commands (defined by overriding the IssueSelector or QuerySelector methods) need be concerned with the selector values of their superclasses. Subclasses of vservicechsr should begin their command selectors with this constant and increment by one for each new command.

In other words, if you add commands to the class-level command space of vservicechsr, you put their implementations in your IssueSelector override and start with the value of vservicechsrNEXT_SELECT as the first selector value you handle.

Subclasses should also provide their own NEXT_SELECT constant for use by their subclasses. This constant has integral type.
