Types

vundoAction

        typedef ... vundoAction;
Represents an undoable action that has been performed by the application. The basic vundoAction does nothing by itself except keep track of whether the action is currently done or undone; applications subclass from this type to do and undo specific tasks by adding more fields and implementing the vundoActAction method. Objects of this type have classes derived from vundoActionClass.

vundoActionClass

        typedef ... vundoActionClass;
Describes the behavior of a class of vundoActions. Note that, unlike most other Galaxy classes, this class does not inherit from vobjectClass (for efficiency reasons). This class is not useful by itself; it must be subclassed in order to do anything.

vundoActionList

        typedef ... vundoActionList;
Represents an action made up of other actions. The default vundoActActionList method for this type simply calls the vundoActAction method of all actions contained by the action list, in appropriate order. Objects of this type have classes derived from vundoActionListClass.

vundoActionListClass

        typedef ... vundoActionListClass;
Describes the behavior of a class of vundoActionList objects. This class inherits from vundoActionClass.

vundoCommand

        typedef ... vundoCommand;
Undo commands are a subclass of vcommand used by vundoLog to represent undoable or redouble functionality.

vundoCommandClass

        typedef ... vundoCommandClass;
Class for undo manager command. Commands of this class have a pointer to their source undo log, and store the text of the last action title.

vundoLog

        typedef ... vundoLog;
Denotes the object that is used to ``record'' all actions that occur in an application with multi-level undo. This object also performs bookkeeping to keep track of which action should be done or undone by selecting the Undo/Redo Menus.

vundoLogClass

        typedef ... vundoLogClass;
Class for vundoLog. Contains overrideable methods to create specialized versions of the undo and redo commands installed by the log.

vundoRedoCommand

        typedef ... vundoRedoCommand;
Redo commands are a subclass of vundoCommand used by vundoLog to represent undoable functionality bound to a symbolic identifier.

vundoRedoCommandClass

        typedef ... vundoRedoCommandClass;
Class representing undo manager Redo commands. This class is used for commands of type vundoRedoCommand.

vundoUndoCommand

        typedef ... vundoUndoCommand;
Undo commands are a subclass of vundoCommand used by vundoLog to represent undoable functionality bound to a symbolic identifier.

vundoUndoCommandClass

        typedef ... vundoUndoCommandClass;
Class representing undo manager Redo commands. This class is used for commands of type vundoUndoCommand.

Functions

vundoActAction

        void vundoActAction(
                vundoAction * action);
Does, undoes, or redoes the undo action action according to the current state of the action. If action is in the NEW state, the action is done. If it is in the DONE state, the action is undone. If it is currently in the UNDONE state, the action is redone. The state of action is also advanced to reflect the action performed. This function is implemented by sending the vundoACT message to the action.

vundoActActionList

        void vundoActActionList(
                vundoActionList * list);
Does, undoes, or redoes the action list list (and all of the sub-actions that are part of the list) according to the current state of the action list. If action is in the NEW state, the action and its sub-actions are done. If it is in the DONE state, the actions are undone. If it is currently in the UNDONE state, the actions are redone. The state of action is also advanced to reflect the action performed.

The sub-actions of the action list are acted in the order they were added to the list if the list is initially in the NEW or DONE state. If the action list is in the UNDONE state, however, the sub-actions are acted in the reverse order that they were added. This function is implemented by sending the vundoACT message to the action.

vundoAddActionListAction

        void vundoAddActionListAction(
                vundoActionList * list,
                vundoAction * action);
Adds action to the action list list, thus making it a sub-action. action then becomes owned by the action list list. The action should be in the DONE state when it is added to the list.

vundoAddLogAction

        void vundoAddLogAction(

Adds action to the undo log log. The action then becomes owned by the log. If any actions in the log are UNDONE when the new action is added, those actions are flushed from the log before adding the new action.

vundoDestroyAction

        void vundoDestroyAction(
                vundoAction * action);
Reclaims action along with any internal storage it uses. action should not be used again after it is passed to this function. This function is implemented by sending the vundoDESTROY message to the action.

vundoDestroyActionList

        void vundoDestroyActionList(
                vundoActionList * list);
Reclaims list along with any internal storage it uses, including all of the sub-actions that have been added to list. The sub-actions are destroyed in the order they were added to the list if the list is currently in the UNDONE state, otherwise they are destroyed in the reverse order they were added. list should not be used again after it is passed to this function. This function is implemented by sending the vundoDESTROY message to the action.

vundoDestroyLog

        void vundoDestroyLog(
                vundoLog * log);
Reclaims log along with any internal storage it uses, including all of the actions that have been added to log. log should not be used again after it is passed to this function.

Note that this does not destroy any commands that have been installed in a command space, because the commands are owned by the command space. You should call vundoUninstallCommands if you are not immediately destroying the command space, so the commands do not try to operate on a non-existent log.

vundoInstallLogCommands

        void vundoInstallLogCommands(
                vundoLog * log,
                vcommandSpace * commandSpace);
Creates an Undo command and a Redo command for the undo log log and installs them in commandSpace under the log's current Undo and Redo command names. These commands call vundoUndoLog or vundoRedoLog (as appropriate) on the log log when issued and return State and Title attributes based on the log's contents when queried.

The undo log remembers where the commands are installed so that when the log's contents change (either due to another action being added, or an Undo or Redo being performed) the changes are propagated from the commands appropriately.

The log also propagates changes in the state of a command with the log's current Save command name in the commandSpace when the save mark for the log changes. Note that no Save command is created and the state is not automatically computed by the commands installed; you must do that yourself. This just ensures that the Save command's state is automatically queried when the undo log's save mark changes.

An undo log can have commands installed in only one command space at a time.

vundoSetActionListName

        void vundoSetActionListName(
                vundoActionList * list,
                const vchar * name);
Sets the name for the given action list. The name is a description of what the action does (for example, ``Paste''). Note that a null pointer can be passed for name if the action has no name.

Unlike ordinary vundoActions, the name is stored on a per-object basis for action lists. This lets you give each action list a different name without creating a new subclass for each list. name is assumed to be a string that is allocated in some global place and is not copied or contained by the action list. (A vname works well for this parameter, although it is not a requirement.)

The initial value of an action list's name is the value of the vundoNAME class variable in the action list's class.

Messages

vundoACT

        <vundoACT>
Does, undoes, or redoes the action according to the current state of action. Note that vundoActAction is implemented by sending this method.

The default method simply advances the state of action (NEW->DONE, DONE->UNDONE, UNDONE->DONE). Subclasses should override this method to do the actual work of the action.

When overriding, you should check the state of the action with vundoGetActionState, and based on that state, either do, undo, or redo the action. If the state is NEW, you should do the action. If it is DONE, you should undo it. If it is UNDONE, you should redo it. You should then call vclassSendSuper to advance the state of the action after you have performed the necessary work.

vundoCREATE_LOG_REDO_COMMAND

        <vundoCREATE_LOG_REDO_COMMAND>
Undo log message which creates the redo command which the undo manager will install. This message is send when vundoCreateLogRedoCommand is called. The default implementation of this message creates a command of the default redo command class. You may override this method if you need to customize the command that the undo manager installs.

vundoCREATE_LOG_UNDO_COMMAND

        <vundoCREATE_LOG_UNDO_COMMAND>
Undo log message which creates the undo command which the undo manager will install. This message is send when vundoCreateLogUndoCommand is called. The default implementation of this message creates a command of the default undo command class. You may override this method if you need to customize the command that the undo manager installs.

vundoGET_NAME

        <vundoGET_NAME>
Returns the name for the action. The name is a description of what the action does (for example, ``Paste''). A null pointer may be returned if the action has no name. Note that vundoGetActionName is implemented by sending this message, and vundoScribeAction calls this indirectly as well.

The default method returns the value of the vundoNAME class variable. You may want to override this function to fetch the name from an object-specific (rather than class-wide) place.

vundoNAME

        <vundoNAME>
This class variable contains the value returned by the default vundoGET_NAME method. You can simply change this class variable to an appropriate name if all names are the same for each action of a class. If the names are different for different actions, you must override the vundoGET_NAME method instead.

vundoNAME can be a null pointer if an action has no name.

Constants

vundoDESTROY

        vundoDESTROY
Destroys action and reclaims any space it uses. Note that vundoDestroyAction is implemented by sending this method.

The default method frees the space occupied by action. Subclasses should override this method if they add any fields to action that contain data that must be destroyed. In order to subclass, first free any data that needs to be freed, and then call vclassSendSuper to free the action itself.
