Types

vapplication

        typedef ... vapplication;
Contains the information needed to represent a single application-objects of this type have classes derived from vapplicationClass. Applications are referenced by a pointer to this type. You can obtain a pointer to your application object with vapplicationGetCurrent. A vapplication contains an attributed object, a set of application attributes, and a client object (provided by the Client class) that manages its global state. Refer to the section ``Application Attributes'' in the ``Application Manager'' chapter of the Galaxy Application Environment Programmer's Guide for a complete description of application attributes. All structure declarations that directly inherit from vapplicationClass must declare a vapplication as their first field so that vapplication pointers can be interchanged with pointers to the new data type.

vapplicationClass

        typedef ... vapplicationClass;
Contains method and variable bindings for objects of type vapplication. Classes declared with this type contain the same set of methods and variables as the default application class, but the behavior of methods and the contents of variables may vary. This class type is derived from vobjectAttributedClass.

vapplicationDependColorFunc

        typedef ... vapplicationDependColorFunc;
Denotes a function which determines dependencies for derived colors for a given application. A function of this type returns a set of color flags that indicate which colors are dependent on the set of color flags flags for the application application.

vapplicationDeriveColorFunc

        typedef ... vapplicationDeriveColorFunc;
Denotes a function which derives colors for a given application. A function of this type returns a color specified by \Codexid based on the known colors known for the application application.

vapplicationNoteProc

        typedef ... vapplicationNoteProc;
Denotes a notification function that is called when an application handles an event of some importance. application is the application handling the event, and event is the event being handled. Importance is determined by the application's event handlers; the default event handler gives no notification. If additional notification is required, you should override the appropriate handler.

vapplicationWindowIterator

        typedef ... vapplicationWindowIterator;
Denotes an abstract index variable used for enumerating an unordered collection of windows that belong to a single application. Typically an application window iterator is used to enumerate all the open windows belonging to an application. Iterators are normally declared as automatic variables of a function and initialized with vapplicationInitWindowIterator.

vapplicationClone

        vapplication * vapplicationClone(
                vapplication * application);
Creates a copy of the application object application and returns a pointer to its contents. Storage for the application object is allocated by the Application class. The Application class reclaims the storage occupied by the application object when it is destroyed with vapplicationDestroy. This function copies all the attributes of the application object except its client object: it will clone any attribute objects that are owned by the application. The application to be cloned can be dynamic or static. This function essentially allocates space for a new application object and calls vapplicationCopyInit on it.

vapplicationCopy

        void vapplicationCopy(
                vapplication * application,
                vapplication * target);
Copies the information in application into target so that the two applications represent equivalent objects. target must have been created or initialized by an Application class function. application and target must have compatible classes for this to work correctly. This function copies all the attributes of the application except its client: it clones any attribute objects that are owned by the application. This function destroys any objects allocated internally by target (including its owned attributes), but does not destroy target itself. The applications to be copied can be dynamic or static. This function is implemented by sending the message vobjectCOPY to application.

vapplicationCopyInit

        void vapplicationCopyInit(
                vapplication * application,
                vapplication * target);
Copies the information in application into target so the memory addressed by target is an application equivalent to application. target should be a pointer to uninitialized memory; the Application class will initialize it as part of the copy. This function copies all the attributes of the application except its client object; it will clone any attribute objects that are owned by the application. This function simply stores the class of application in target and sends the message vobjectCOPY_INIT to application.

vapplicationCreate

        vapplication * vapplicationCreate();
Creates a new application object of the default application class and returns a pointer to its contents. Storage for the application object is allocated by the Application class. The Application class will reclaim the storage occupied by the application object when it is destroyed with vapplicationDestroy. The new application object is created with a set of default attributes, as determined by the default application class. The default attributes of the default application class are documented in the ``Application Manager'' chapter of the Galaxy Application Environment Programmer's Guide. This function calls the Client class function vclientCreate to create a client object for the application object. This function essentially allocates space for a new application object and calls vapplicationInit on it.

vapplicationDestroy

        void vapplicationDestroy(
                vapplication * application);
Destroys the application object represented by application, which must have been created or initialized by an Application class function. Use this function to reclaim the storage occupied by an application. The application should not be referenced by any other object when it is destroyed. This function only attempts to free the storage occupied by application if it was allocated by vapplicationCreate, vapplicationCreateOfClass, or vapplicationClone. This function will also destroy any attribute objects that are owned by the application. This function calls vclientDestroy to destroy the application's global state. This function is implemented by sending the message vobjectDESTROY to application.

vapplicationDetermineColor

        vcolor * vapplicationDetermineColor(
                vapplication * application,
                int id);
Derives a color that the windows of the application application use to draw their contents. The identifier id specifies the function of the color in the application color scheme; it should be one of the functional color identifiers provided by the Drawing class (vdrawCOLOR_SHADOW_TOP, vdrawCOLOR_HILITE, for example). The color is derived from the color attributes of the application and its class. This function never returns NULL.

vapplicationDetermineColorScheme

        void vapplicationDetermineColorScheme(
                vapplication * application,
                unsigned long flags,
                vcolorScheme * scheme);
Derives a color scheme that the windows of the application application use to draw their contents. flags is a set of Color class color flags that specifies which colors to derive. The colors are derived from the color attributes of the application and its class. The derived colors are returned in the color scheme scheme. Calling this function for multiple colors is more efficient than deriving the colors individually with vapplicationDetermineColor.

vapplicationDetermineColorSchemeFrom

        void vapplicationDetermineColorSchemeFrom(
                vapplication * application,
                vcolorScheme * known,
                unsigned long flags,
                vcolorScheme * scheme);
Derives a color scheme that the windows of the application application use to draw their contents. flags is a set of Color class color flags that specifies which colors to derive. The colors are derived from the color attributes of the application and its class as well as the colors provided in the color scheme known. The derived colors are returned in the color scheme scheme. Calling this function for multiple colors is more efficient than deriving the colors individually with vapplicationDetermineColor. Colors in the scheme of known colors are given precedence over those defined by the application and its class.

vapplicationDetermineColors

        void vapplicationDetermineColors(
                vapplication * application,
                int * ids,
                int count,
                vcolor ** colors);
Derives a set of colors that the windows of the application application use to draw their contents. The identifiers in ids specify the function of the colors in the application's color scheme; they should be comprised of the functional color identifiers provided by the Drawing class (vdrawCOLOR_SHADOW_TOP, vdrawCOLOR_HILITE, for example).

The colors are derived from the color attributes of the application and its class. The derived colors are returned in the array elements of colors corresponding to the array elements of ids. There are count identifiers in the array ids, and count colors are returned in the array colors. This function never returns a NULL color. Calling this function for multiple colors is more efficient than deriving the colors individually with vapplicationDetermineColor.

vapplicationDetermineColorsFrom

        void vapplicationDetermineColorsFrom(
                vapplication * application,
                long flags,
                vcolor ** known,
                int * ids,
                int count,
                vcolor ** colors);
Derives a set of colors that the windows of the application application use to draw their contents. The identifiers in ids specify the function of the colors in the application's color scheme; they should be comprised of the functional color identifiers provided by the Drawing class (vdrawCOLOR_SHADOW_TOP, vdrawCOLOR_HILITE, for example).

The colors are derived from the color attributes of the application and its class as well as the colors provided in the array known. The derived colors are returned in the array elements of colors corresponding to the array elements of ids. There are count identifiers in the array ids, and count colors are returned in the array colors. This function never returns a NULL color. The array known should point to an array of at least vdrawCOLOR_MAX color pointer elements. flags identifies which colors in the array are present at the index corresponding to the bit index of the flag value. For example, 1L<<vdrawCOLOR_SHADOW_TOP indicates that a background color can be found at known[vdrawCOLOR_SHADOW_TOP]. Colors in the array of known colors are given precedence over those defined by the application and its class.

vapplicationLoad

        vapplication * vapplicationLoad(
                vresource resource);
Reads an application from the resource resource and returns a pointer to its contents. Storage for the application object is allocated by the Application class. The application's class is determined by the value of the tag ``Class.'' It generates vobjectUndefinedClassException if the name does not denote a class in the Class class registry. Note that this function creates an object based on the class of the resource, even if this class is not an application class. The object pointer is cast to an application pointer, even if the object represents a different data type. Attributes may be initialized based on the contents of the resource, or they may be set to default values, depending on the class.

The attributes of the default application class, which are stored in resources, are documented in the ``Application Manager'' chapter of the Galaxy Application Environment Programmer's Guide. This function essentially allocates space for a new application and calls vapplicationLoadInit on it.

vapplicationNextWindowIterator

        int vapplicationNextWindowIterator(
                vapplicationWindowIterator * iterator);
Performs the next step of the iteration represented by iterator and returns TRUE if a window is available, FALSE if not. The particular window can be retrieved from the iterator with vapplicationGetWindowIteratorWindow. For example, the following code fragment repaints every window belonging to the current application: 


vapplicationWindowIterator iterator;
vapplicationInitWindowIterator(&iterator, vapplicationGetCurrent());
while (vapplicationNextWindowIterator(&iterator))                      
  	           
vwindowInval(vapplicationGetWindowIteratorWindow(&iterator), 0);
vapplicationDestroyWindowIterator(&iterator);

Notifies the application application that the event event was handled by the application's event handler. This function is implemented by sending the message vapplicationNOTIFY to application.

vapplicationPropagateChange

        void vapplicationPropagateChange(

vapplicationSET_PREFERENCES

        <vapplicationSET_PREFERENCES>
Changes the preferences for the application application to resource. The default preferences of an application are initialized to the application's preference dictionary (named appname), located in the user resource file (.vgalaxy.vr). The function entry-point vapplicationSetPreferences is implemented by sending this message to the application.

The default method for the default application class stores the resources in the application structure and rebuilds the preference database using the new application preferences. Subclasses that override this method should first call vclassSendSuper to set the application preferences, then update any subclass-specific state.

vapplicationSET_SPACE

        <vapplicationSET_SPACE>
Changes the instance command space of the application application to space. space can be NULL, in which case the application is given no instance command bindings. Bindings in the instance command space take precedence over any bindings in the application's class command space. The application is added as a new owner of the command space, and is removed from ownership of its previous command space, if any. Any changes to attribute bindings are automatically propagated by this message. Refer to the ``Command Manager'' chapter of the Galaxy Application Environment Programmer's Guide for more information about command spaces.

The function entry-point vapplicationSetSpace is implemented by sending this message to the application. The default method for the default application class stores the command space reference in the application structure, clears the command space ownership flag, and propagates any necessary changes to the Command class. Subclasses that override this method should first call vclassSendSuper to set the application command space, then update any subclass-specific state.

vapplicationNEXT_ID_ATTRIBUTE

        vapplicationNEXT_ID_ATTRIBUTE
Denotes the next available attribute ID for subclasses of the default application class. This constant is provided so that attribute IDs defined by subclasses do not conflict with those defined by the Application class. Only attributes defined by overriding the methods vobjectSET_ID_ATTRIBUTE and vobjectGET_ID_ATTRIBUTE need be concerned with the attribute IDs of their superclasses. Subclasses of vapplicationClass should begin their attribute IDs with this constant and increment by one for each attribute. Subclasses should also provide their own NEXT_ID_ATTRIBUTE constant for use by their subclasses. This constant has integral type.

vapplicationNEXT_SELECT

        vapplicationNEXT_SELECT
Denotes the first command selector available to subclasses of the default application class. This constant is provided so that selector values defined by subclasses do not conflict with those defined by their superclasses. Only class-level commands (defined by overriding the methods vobjectISSUE_SELECTOR and vobjectQUERY_SELECTOR) need be concerned with the selector values of their superclasses. Subclasses of vapplicationClass should begin their command selectors with this constant and increment by one for each new command. In other words, if you add commands to the class-level command space of vapplication, you put their implementations in your vobjectISSUE_SELECTOR override and start with the value of vapplicationNEXT_SELECT as the first selector value you handle. Subclasses should also provide their own NEXT_SELECT constant for use by their subclasses. This constant has integral type.

Macros

vapplicationCLASS

        #define vapplicationCLASS(...) ...
Contains the method and variable declarations for application classes. This macro expands to the contents of vapplicationClass. SUPER is the type identifier of the class being declared (for example, vapplicationClass). This type is used by the class's subclasses to refer to it.
