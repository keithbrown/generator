Types

vsession

        typedef ... vsession;
Contains the information needed to maintain a session between a service consumer and a service provider. Both the service consumer and the service provider have a pointer to a vsession that is used as the application level interface between the two applications.

The service consumer's vsession is created by the user's application with vsessionCreate. It must also be explicitly destroyed by the user's application with vsessionDestroy. However, ownership may be passed to the Session class to destroy it with vsessionSetDestroyWhenInactive when the session ends. The service consumer must explicitly begin a session with vsessionBegin.

The service provider's vsession is created, started, and maintained by the Session class itself. These sessions are derived from the template session for the service. The template session can be configured using the result of vserviceGetTemplateSession. By default, the service provider's session is destroyed by the Session class when the session with the service consumer ends. The Service class begins the sessions it creates and then notifies the session once it is ready. The application can set the notification function on the template session to become aware of these generated sessions.

The service provider or the service consumer, may terminate a session by calling vsessionEnd.

vsessionAttributeIterator

        typedef ... vsessionAttributeIterator;
Denotes an abstract variable used to sequence through the attributes for a session. The iterator contains vname/vdatatag/value triplets. The type of value is determined by the vdatatag. An iterator becomes invalid if any session attributes are added, removed, or modified. Iterators are normally declared as automatic variables and are initialized with vsessionInitAttributeIterator.

vsessionClass

        typedef ... vsessionClass;
Contains method and variable bindings for objects of type vsession. Classes declared with this type contain the same set of methods and variables as the default session class, but the behavior of methods and the contents of variables can vary. This class type is derived from vloadableClass.

vsessionEvent

        typedef ... vsessionEvent;
Denotes an event that is passed to the vsessionNoteProc. The function vsessionGetEventCode is used to get the code categorizing the event.

vsessionEventCode

        typedef ... vsessionEventCode;
An integral type that categorizes an event sent to a session notify function. The code is retrieved from the event with vsessionGetEventCode. It can be one of vsessionEVENT_BEGIN, vsessionEVENT_CANNOT_BEGIN, or vsessionEVENT_END.

vsessionNoteProc

        typedef ... vsessionNoteProc;
Denotes a notification function that is called when a session is handling an event, such as when the session starts or ends. session is the session handling the event, and event is the event being handled. vsessionGetEventCode can be called to determine the type of event.

vsessionStatement

        typedef ... vsessionStatement;
Denotes a message that an application sends to a service through a session. The vsessionStatement is only used by service consumers, whereas the vservicePrimitive is only used by service providers.

Statements are either created with vsessionCreateStatement, or initialized with vsessionInitStatement. Statements are destroyed by calling vsessionDestroyStatement. Statements are sent to a service provider by sending them through the session with vsessionSendStatement.

vsessionStatementClass

        typedef ... vsessionStatementClass;

vsessionStatementEvent

        typedef ... vsessionStatementEvent;
Denotes an event that is being handled by a vsessionStatement. The function vsessionGetStatementEventCode can be used to get the code categorizing the event.

vsessionStatementEventCode

        typedef ... vsessionStatementEventCode;
An integral type that categorizes an event sent to a statement notify function. The code is retrieved from the event with vsessionGetStatementEventCode. It can be one of vsessionStatementEVENT_COMPLETED, vsessionStatementEVENT_CANNOT_SEND, vsessionStatementEVENT_END_SESSION, or vsessionStatementEVENT_FAILED.

vsessionStatementNoteProc

        typedef ... vsessionStatementNoteProc;
Denotes an observationa notification function that is called when a statement is handling an event, such as when the outcome of sending the statement is known. statement is the statement handling the event, and event is the event being handled. vsessionGetStatementEventCode can be called to determine the type of event.

vsessionStatementStatus

        typedef ... vsessionStatementStatus;
Denotes the last known status of a statement. If a statement is still pending, its outcome may already have been determined, but is not yet be known to the statement. A statement's status can be checked by calling vsessionGetStatementEventCode. Possible status values are:
  • vsessionStatementSTATUS_UNSENT
  • vsessionStatementSTATUS_PENDING
  • vsessionStatementSTATUS_COMPLETED
  • vsessionStatementSTATUS_FAILED
  • vsessionStatementSTATUS_ENDED_SESSION.

Functions

vsessionAddObject

        void vsessionAddObject(
                vsession * session,
                const vname * name,
                vobject * object);
Gives the session session access to the object object, using the name name. This function is only intended for service providers that are using the default handler for the primitive returned by vserviceCreateIssueCommandPrimitive, however the application can create other primitives that make use of this facility. Objects that are globally shared by all sessions for the service can be added to the service with vserviceAddObject instead of to each session.

Ownership of the object is still retained by the application. The object should be removed with vsessionRemoveObject before the object is destroyed.

vsessionBegin

        void vsessionBegin(
                vsession * session);
Attempts to start a new session with the service described by the session session. If the session has a notification function, the session is established asynchronously, notifying the session with vsessionEVENT_BEGIN when it is ready, or vsessionEVENT_CANNOT_BEGIN if the session could not be started. If the session does not have a notification function, the session is established synchronously and is ready when the function returns, or generates an exception if the session could not be started properly. Normally sessions have notification functions.

While the session is being established asynchronously, vsessionIsActivating is TRUE. Any statements sent are queued until the session is established, and are flushed when the session is ready, after it is notified. When the session is ready, vsessionIsActive is TRUE.

Service providers do not need to explicitly call vsessionBegin, since the Service class automatically does this when it generates the session with the service consumer.

Either side may terminate a session at any time by calling vsessionEnd. Destroying the session with vsessionDestroy or exiting the application also causes the session to terminate, but the other end of the session may take an arbitrarily long time to be notified depending on the underlying protocols that the session is using. This call has no effect if the session is already active or activating.

vsessionBlockStatement

        int vsessionBlockStatement(
                vsessionStatement * statement,
                vtimestamp * timeout);
Blocks the application until the statement statement is notified, or the timestamp specified by timeout expires. If timeout is NULL then the application blocks until the statement is notified. If the specified time is of duration 0,0 then the network is polled to see if the result is waiting.

The application might process other events coming from the session that the statement was sent to while it is blocking, however it may queue events coming from other sources, such as other sessions, GUI objects, and timers.

The statement is notified before vsessionBlockStatement returns, unless the timeout expires. A result of FALSE means that timeout expired before the statement was notified.

If vsessionGetDestroyWhenFinished is TRUE when the statement notification function returns, then statement is invalid when this function returns. This function has no effect if statement is not pending.

vsessionCall

        vsessionStatementStatus vsessionCall(
                vsession * session,
                vtimestamp * timeout,
                vsignature * sig,
                ... arg4);
Encapsulates an entire synchronous remote invocation. This function calls the operation indicated by the given signature using the given arguments and waits for results for the specified time. The variable argument list passed to this function contains storage for the expected return value (if any) followed by variables indicating the arguments to the remote operation.

vsessionClone

        vsession * vsessionClone(
                vsession * session);
Creates a copy of the session session and returns a pointer to it. The cloned session is not active, even if session is active. It does not belong to a service even if session belongs to a service. It does, however, copy attributes and added objects.

The caller is responsible for eventually calling vsessionDestroy to reclaim memory when the clone is no longer needed.

vsessionCopyInit

        void vsessionCopyInit(
                vsession * sessionSource,
                vsession * sessionDest);
Initializes the session sessionDest as a copy of the session sessionSource. The resulting copy is not active, even if the source was. It does not belong to a service even if the source does. It does, however, copy attributes and added objects.

The caller is responsible for eventually calling vsessionDestroy to reclaim memory when the copy is no longer needed.

vsessionCreate

        vsession * vsessionCreate();
Allocates and initializes a new session and returns a pointer to it. The resulting session is not active, nor does it have a service.

The result can be used to start a session with a service provider by adding individual attributes with vsessionSetAttribute or sets of attributes with vsessionSetAttributesFromScrap, and calling vsessionBegin.

The caller is responsible for eventually calling vsessionDestroy to reclaim memory when the session is no longer needed. Alternatively, the Session class can destroy the session when the session terminates by calling vsessionSetDestroyWhenInactive with TRUE.

This call is intended for service consumers. Service providers automatically get their session instances from the Service class.

vsessionCreateOfClass

        vsession * vsessionCreateOfClass(
                vsessionClass * clas);
Creates a new session of class clas and returns a pointer to the new session. Memory is allocated by the Session class. The new session is created with a set of default attributes, determined by the class.

vsessionCreateStatement

        vsessionStatement * vsessionCreateStatement();
Allocates and initializes a new statement and returns a pointer to it. The statement must be given a signature with vsessionSetStatementSignature and a session with vsessionSetStatementSession before the arguments can be set. Arguments are set using vsessionSetStatementArgs.

The statement should be destroyed with vsessionDestroyStatement to reclaim memory when it is no longer needed. Alternatively, the Session class can destroy the statement when the statement outcome is known by calling vsessionSetDestroyStatementWhenFinished with TRUE.

vsessionDestroy

        void vsessionDestroy(
                vsession * session);
Reclaims all memory used by the session session and invalidates the pointer. Active sessions should be terminated by calling vsessionEnd before calling vsessionDestroy or risk the chance of any data it has sent across the session not arriving. vsessionSetDestroyWhenInactive should be used when destroying a session from within a notification function.

This function generates an exception if the result of vsessionGetDestroyWhenInactive is TRUE since that means that the Session class has ownership.

vsessionDestroyAttributeIterator

        void vsessionDestroyAttributeIterator(
                vsessionAttributeIterator * iterator);
Terminates the iteration represented by the iterator iterator, even if it is not yet exhausted. This function is necessary to reclaim internal iteration objects allocated by the Session class.

vsessionDestroyStatement

        void vsessionDestroyStatement(
                vsessionStatement * statement);
Reclaims all memory used by the statement statement and invalidates the pointer. If the statement has been sent but has not yet finished, then any further events and results are discarded if and when they arrive. vsessionSetDestroyStatementWhenFinished should be used if a statement is destroyed within a notification function. The statement signature is not affected.

vsessionDump

        void vsessionDump(
                vsession * session,
                FILE * fp);
Dumps information about the session session to the stream fp. If fp is NULL then vdebugGetInfoStream is used. This function is primarily used for debugging. It is always defined, but does not produce any output when vportDEBUG is FALSE.

vsessionEnd

        void vsessionEnd(
                vsession * session);
Attempts to terminate the session session. If the session has a notification function, the session is terminated asynchronously, notifying the session with vsessionEVENT_END when it finishes terminating. If the session does not have a notification function, the session is terminated synchronously. Either end of the session may terminate it. Both ends receive the event when the session is actually terminated.

Since notification may occur before the function returns, session may be invalid if vsessionGetDestroyWhenInactive is TRUE when the function returns. Otherwise, the session is deactivating until the session is actually terminated. This function has no effect if the session is already inactive.

vsessionGetIssueCommandSignature

        vsignature * vsessionGetIssueCommandSignature();
Returns a signature which is used for a service consumer to issue a command to a particular object in a service provider.

The signature takes three arguments. The first argument is a vstr string denoting the name of the object that the command should be issued to. The second argument is a vname name denoting the name of the command to issue. The last argument is a vscrap dictionary scrap denoting the context dictionary. The signature return result is a vbool which is TRUE if the command was handled and FALSE otherwise.

For service consumers creating statements, the dictionary scrap can be created using vdatatagScrapFromValue with a dictionary datatag. This datatag describes a vdict which is used as the context dictionary.

For service providers writing a primitive handler, the dictionary scrap can be converted to the vdict context dictionary using vdatatagScrapFromReference with a dictionary datatag that describes the expected contents.

The Session class provides the convenience function vsessionMakeIssueCommandStatementScribed for sending statements, and the Service class provides the convenience function vserviceMakeIssueCommandPrimitive for implementing the primitive. The resulting signature should not be destroyed by the application.

vsessionSendStatement

        void vsessionSendStatement(
                vsessionStatement * statement);
Sends the statement statement to the session that is returned by vsessionGetStatementSession. An exception is generated if the session was not set with vsessionSetStatementSession, the signature was not set with vsessionSetStatementSignature, or the arguments were not set with vsessionSetStatementArgs or vsessionSetStatementArgsScrap.

The statement notification function is called when the result of the statement is known. This may actually occur before vsessionSendStatement returns. If the statement does not have a notification function, then the return result is available when the statement's status is vsessionStatmentSTATUS_COMPLETED. Note that if vsessionSetDestroyStatementWhenFinished is TRUE, and there is no notification function set, the statement is destroyed when the return result becomes available, and the return result is discarded.

When the statement has completed, the return result, if any, can be retrieved with vsessionGetStatementReturnValue.

If the statement is destroyed before the outcome is known it is still sent to the provider as normal, but any return results and notification are discarded.

vsessionSetAttribute

        void vsessionSetAttribute(
                vsession * session,
                const vname * name,
                vdatatag * datatag,

vsessionSetDestroyStatementWhenFinished

        void vsessionSetDestroyStatementWhenFinished(
                vsessionStatement * statement,
                int bOn);
Tells the Session class whether the statement statement should be destroyed after it is notified with an event that causes the statement to no longer be pending. If bOn is TRUE, the statement is destroyed after the event, if bOn is FALSE, the statement must be explicitly destroyed by the application. The default value is FALSE.

By setting the value to TRUE, the application effectively passes ownership of the statement to the Session class, and grabs ownership back by setting the value to FALSE.

This function can be called at any time, regardless of whether or not the statement is currently pending, but the statement is only destroyed after it handles an event that makes its status no longer pending. If the statement has a notification function, the statement is not destroyed until after the notification function returns. The decision to destroy the statement is not determined until after the notification function returns.

vsessionSetDestroyWhenInactive

        void vsessionSetDestroyWhenInactive(
                vsession * session,
                int bOn);
Tells the Session class whether the session session should be destroyed after the vsessionEVENT_END event is handled in the notification function. If bOn is TRUE, the session is destroyed after the event; if bOn is FALSE, the session must be explicitly destroyed by the application.

For service providers, the default value is TRUE because the Service class creates and manages the session. For service consumers, the default value is FALSE because the application creates and manages the session.

By setting the value to TRUE, the application effectively passes ownership of the session to the Session class, and grabs ownership back by setting the value to FALSE. This function can be called at any time, regardless of whether or not the session is active.

Messages

vsessionBEGIN

        <vsessionBEGIN>
Attempts to start a new session with the service described by the session session. If the session has a notification function, the session is established asynchronously, notifying the session with vsessionEVENT_BEGIN when it is ready, or vsessionEVENT_CANNOT_BEGIN if the session could not be started. If the session does not have a notification function, the session is established synchronously and is ready when the function returns, or generates an exception if the session could not be started properly. Normally sessions have notification functions.

While the session is being established asynchronously, vsessionIsActivating is TRUE. Any statements sent are queued until the session is established, and are flushed when the session is ready, after it is notified. When the session is ready, vsessionIsActive is TRUE.

Service providers do not need to explicitly call vsessionBegin, since the Service class automatically does this when it generates the session with the service consumer.

Either side may terminate a session at any time by calling vsessionEnd. Destroying the session with vsessionDestroy or exiting the application also causes the session to terminate, but the other end of the session may take an arbitrarily long time to be notified depending on the underlying protocols that the session is using. This call has no effect if the session is already active or activating.

vsessionBLOCK

        <vsessionBLOCK>
Wait for activity on a session for the period of time specified by timeout. If timeoutNULL this method blocks indefinitely. Activity is defined as an control or i/o event on the underlying transport, for example data arrival, write-side flow control, or connection loss.

vsessionCALL

        <vsessionCALL>
Encapsulates an entire synchronous remote invocation. This method calls the operation indicated by the given signature using the given arguments and waits for results for the specified time. The variable argument list passed to this method contains storage for the expected return value (if any) followed by variables indicating the arguments to the remote operation.

vsessionEND

        <vsessionEND>
Attempts to terminate the session session. If the session has a notification function, the session is terminated asynchronously, notifying the session with vsessionEVENT_END when it finishes terminating. If the session does not have a notification function, the session is terminated synchronously. Either end of the session may terminate it. Both ends receive the event when the session is actually terminated.

Since notification may occur before the function returns, session may be invalid if vsessionGetDestroyWhenInactive is TRUE when the function returns. Otherwise, the session is deactivating until the session is actually terminated. This function has no effect if the session is already inactive.

vsessionNOTIFY

        <vsessionNOTIFY>

vsessionNOTIFY_STATEMENT

        <vsessionNOTIFY_STATEMENT>

vsessionSEND_STATEMENT

        <vsessionSEND_STATEMENT>
Sends the statement statement over a session. This method only adds the statement the the queue of statements to be sent by a session; the statement is not necessarily sent immediately.

vsessionSTARTED

        <vsessionSTARTED>
Notifies the session that it has completely started. This method is used by session subclasses to insert additional processing steps into the session startup sequence. The default implementation of this method sends vsessionEVENT_BEGIN to the session.

vsessionTERMINATED

        <vsessionTERMINATED>
Notifies a session that it has terminated completely. This method is used by session subclasses to insert additional processing steps into the session teardown sequence. The default implementation of this method notifies all outstanding statements that the session has terminated, and then sends vsessionEVENT_END to the session.

Constants

vsessionEVENT_BEGIN

        vsessionEVENT_BEGIN
Denotes that a session has just started. When a service consumer gets this event, the session parameter is the session that was passed to vsessionBegin. When a service provider gets this event, the session parameter to the observationnotification function is a new session derived from the result of vserviceGetTemplateSession. The service that the session is bound to can be retrieved with vsessionGetService.

Any statements queued to be written to the session (because they were sent while the session was still activating) are flushed after the observationnotification function is handled.

vsessionEVENT_CANNOT_BEGIN

        vsessionEVENT_CANNOT_BEGIN
Denotes that vsessionBegin could not succeed. The session parameter is the session that was passed to vsessionBegin. The session is still valid, but is destroyed after the observationnotification function returns if vsessionGetDestroyWhenInactive is TRUE.

vsessionEVENT_END

        vsessionEVENT_END
Denotes that a session has just ended. The session parameter is the session that was passed to the vsessionEVENT_BEGIN event that marked the start of the session.

If vsessionGetDestroyWhenInactive is TRUE, the session is destroyed after the observationnotification function returns. This is the default for service providers, but not for service consumers.

The vsessionEVENT_END notification is sent after either application in the session calls vsessionEnd, or when the Session class detects that the other application is no longer accessible.

Any statements still pending are notified with vsessionStatementEVENT_END_SESSION and their status changed to vsessionStatementSTATUS_ENDED_SESSION.

vsessionEVENT_LOCAL_EXCEPTION

        vsessionEVENT_LOCAL_EXCEPTION
Denotes that an exception occurred while managing the local side of a session. The session parameter is the session that passed to the vsessionEVENT_BEGIN event that marked the exception occurrance.

Since sessions can operate asynchronously and thus throw exceptions asynchronously, the Session class traps any asynchronous exceptions and converts them to notifications so that the context in which the exception occurred is preserved for use by the application.
