vlist

        typedef ... vlist;
Denotes an instance of the class derived from vlistClass. This type is useful for making any two dimensional data type where transformations on the data are common. You can also use this type as the basis for new types of lists tailored to custom needs.

vlistCellValueGetterProc

        typedef ... vlistCellValueGetterProc;
Denotes a list CellValueGetter procedure. To use your own representation for the data stored in a list, write a function to match this prototype. row, column denote the cell whose value is being retrieved. When the cell specified by row and column are outside the range of the list, a /Arg/Range exception should be generated.

vlistCellValueSetterProc

        typedef ... vlistCellValueSetterProc;
Denotes a list CellValueSetter procedure. To use your own representation for the data stored in a list, write a function to match this prototype. row, column denote the cell whose value is being set. value is the value to put in the cell. When the cell specified by row and column are outside the range of the list list, the list should be expanded to contain the cell.

vlistCellValueSorterFunc

        typedef ... vlistCellValueSorterFunc;
A function which compares two cell values. first and second are cell values returned by the CellValueGetter procedure for the cells being compared. The function should return a value less than zero if first < second, a value greater than zero if first > second, and zero if first == second.

vlistClass

        typedef ... vlistClass;
The class definition for the list object, it is derived from vobjectClass. It contains the vobjectClass members plus the message definitions for list manipulation.

vlistIterator

        typedef ... vlistIterator;
This type is used to move over a list. An iterator can be used to move over the entire list, or just a selection of a list. When an iterator is working on a selection, it remembers the selection internally, so it is safe to alter the selection without disturbing the iterator. Changing cell values is also legal while an iterator is working on the list, however it is not safe to add/remove/rearrange rows or columns to the list while the iterator is being used.

vlistIteratorClass

        typedef ... vlistIteratorClass;
Denotes the list iterator class. List iterator objects are used to move over a list

vlistNameCellProc

        typedef ... vlistNameCellProc;
Denotes a function that returns the name of the cell denoted by row, column. The name is used internally when the list is sorted or searched. You would write your own version when the default behavior of returning the data value for the cell is not proper. When the cell specified by row and column are outside the range of the list list, the exception /Arg/Range should be raised.

vlistObserver

        typedef ... vlistObserver;
This type is a used are mainly for synchronizing a list with a visual representation in a dialog. Observers are registered with one list (a list can have multiple observers associated with it), and is notified when actions happen to the list. Another use of the observers is to track operations done to a list so as to maintain an undo log. (Refer to ``Undo Manager'' in the Galaxy Application Environment Programmer's Guide for details.)

vlistCopy

        void vlistCopy(
                vlist * list,
                vlist * target);
Copies the information from list into target, so that the two list objects represent equivalent list objects but separate storage. Any objects which exist in list are duplicated in target. The cell values are also copied by calling vlistGetCellValue and vlistSetCellValue over the contents of the list (this means the pointers are duplicated, not the memory they point to).

\noteSelections and observers are not copied. The pointers in the object's dataset and data pointer field are copied, but the memory they point to is not duplicated. This function is implemented by sending the message vobjectCOPY to list.

vlistCopyInit

        void vlistCopyInit(
                vlist * list,
                vlist * target);
Copies the information in list into target so the memory addressed by target is a list equivalent to list. target should be a pointer to uninitialized memory---the List Manager initializes it as part of the copy. This function copies all the attributes of the list, and it clones the list contents by calling vlistGetCellValue for every cell in list and subsequently calling vlistSetCellValue using target. This function is implemented by sending the message vobjectCOPY_INIT to list.

void exampleSelectionReport(vlistview *listview)
{
vlistIterator        iterator;
int                        row, column;
vlistInitSelectionIterator(&iterator, vlistviewGetSelection(listview));
while (vlistNextIterator(&iterator)) {
        row = vlistGetIteratorRow(&iterator);
        column = vlistGetIteratorColumn(&iterator);

        }

} /* exampleSelectionReport */

vlistFindCell

        int vlistFindCell(
                vlist * list,
                const vchar * name,
                int * row,
                int * column,
                int startRow,
                int startColumn,
                int directionHint);
Searches the list list starting at the cell identified by startRow, startColumn. A match is found when name matches the cell's name (vlistDetermineCellName) for the length of name, the comparison is case insensitive. The search works in the direction specified by directionHint, directionHint can be vlistFIND_COLUMN_MAJOR or vlistFIND_ROW_MAJOR. vlistFIND_COLUMN_MAJOR means that all columns in a row are searched before moving to the next row, and vlistFIND_ROW_MAJOR means all rows are checked before moving to the next column (refer to Figure). If the list has been sorted, then the find operation can be achieved more quickly by bitwise ORing (|) vlistFIND_LIST_IS_SORTED into directionHint.

Figure: The search methods of a list find

A match is defined to be when name matches (vcharCompare) with a cell name (retrieved with vlistDetermineCellName).

When the match is found, its location is returned through row and column, and TRUE is returned. If no match is found, then zero is placed in row and column, and FALSE is returned.

The search is carried out by the vlistFIND_CELL method of the list. When a custom data representation is being used that allows for better searching, this method may also be worth overriding.

vlistGetOccurrenceColumn

        int vlistGetOccurrenceColumn(
                vlistOccurrence * occurrence);
The return value is only of interest for the following types of list occurrences:
  • vlistCELL_VALUE_CHANGE_OCCURRENCE
  • vlistINSERT_COLUMN_RANGE_OCCURRENCE
  • vlistREMOVE_COLUMN_RANGE_OCCURRENCE
  • vlistSWAP_COLUMN_RANGE_OCCURRENCE
  • vlistMOVE_COLUMN_RANGE_OCCURRENCE
For the occurrence types listed, the column indicates a column involved in the occurrence. For insertions and moves, the column is the one before which the insertion takes place; for removals, the column is the first column in the the range removed. For swaps the column is the first column in the second range being swapped.

The type of a list occurrence can be determined by a call to vlistOccurrenceGetType.

For the occurrence types listed, the column count indicates a number of columns involved in the occurrence. For insertions, removals, swaps and moves, the column count is the number of columns being inserted, removed, swapped or moved. For count changes, the column count is the total number of columns in the list after the action that caused the count change.

The type of a list occurrence can be determined by a call to vlistOccurrenceGetType.

vlistLoad

        vlist * vlistLoad(
                vresource resource);
Reads a list data object from the resource resource and returns a pointer to the newly created object, space for the list is allocated by the List Manager. An exception is generated if the resource has not been constructed properly, or if the class in the resource is not defined .

vlistLoad retrieves values from resource and sets the row count, the column count, and data storage orientation. The default implementation does not load cell contents, but provides a framework for list cell resource loading.

If the list is not empty (that is, the row count and column count are greater than zero) the List Manager retrieves from resource the resource that was created by vlistMakeMajorResource during list storage. The major list storage axis (defined by vlistSetOrientation) is iterated over. vlistFindMinorResource is called to find the resource that actually contain the list cell storage. vlistLoadCell is finally called to actually get the value from the resource and set the cell value. If vlistFindMinorResource returns vresourceNULL, then the loading for that particular row or column is skipped.

More concretely, suppose that the list's orientation is vlistCOLUMN_MAJOR; vlistLoad does this: 


vresource minorResource;

/* majorResource has already been determined */

for (column = 0; column < vlistGetColumnCount(list); column++)
 { 
 minorResource = vlistFindMinorResource (list, majorResource, column);
 if (minorResource != vresourceNULL)
  {
  for (row = 0; row < vlistGetRowCount(list); row++)
   { 
   vlistLoadCell (list, row, column, minorResource);
   }
  }
 }

By default, vlistFindMinorResource returns vresourceNULL, and vlistLoadCell does nothing.

        void vlistSetOrientation(
                vlist * list,
                int direction);
Changes the data storage orientation for the list list to be direction. Values for direction can be vlistROW_MAJOR or vlistCOLUMN_MAJOR.

The orientation is a hint to the List Manager that manipulation operations in a particular dimension are more prevalent than in the other.         void vlistSort(
                vlist * list,
                int axis,
                int key,
                vlistCellValueSorterFunc compar);
Sorts the list list. axis denotes the part of the list to exchange during sorting. If axis is vlistROW, then rows are exchanged. vlistStore

        void vlistStore(
                vlist * list,
                vresource resource);
Writes the contents of the list object list into the resource resource. The resource resource should be an untyped resource created with vresourceMake. The list object can be recovered from the resource at a later time with vlistLoad or vlistLoadInit.

vlistStore stores values such as the row count, column count, and orientation. The default implementation does not actually store the list contents, but provides a framework for list cell storage:

If the list is not empty (that is, the row count and column count are greater than zero) the List Manager calls vlistMakeMajorResource so that a resource repository can be created inside of resource. The cell contents are iterated over along the major storage axis as set by vlistSetOrientation. For each unit along the major axis (for each column in a list with an orientation of vlistCOLUMN_MAJOR), vlistMakeMinorResource is called. Each cell along this axis is iterated over, having vlistStoreCell called on it. If vlistMakeMajorResource returns vresourceNULL, the list contents are not stored. If vlistMakeMinorResource returns vresourceNULL, that particular unit's contents are not stored.

More concretely, suppose the list's orientation is vlistCOLUMN_MAJOR, vlistStore will do this: 


vresource majorResource, minorResource;
majorResource = vlistMakeMajorResource (list, storageResource);
if (majorResource != vresourceNULL) {
    for (column = 0; column < vlistGetColumnCount(list); column++) {
        minorResource = vlistMakeMinorResource (list, majorResource, column);
        if (minorResource != vresourceNULL) {
        for (row = 0; row < vlistGetRowCount(list); row++) {
                        vlistStoreCell (list, row, column, minorResource;
                 }
        }
    }
}

By default, vlistMakeMajorResource and vlistMakeMinorResource return vresourceNULL, and vlistStoreCell does nothing.

        <vlistFIND_CELL>
Searches the list list starting at the cell identified by startRow, startColumn. The search works in the direction specified by directionHint, directionHint can be vlistFIND_COLUMN_MAJOR or vlistFIND_ROW_MAJOR. vlistFIND_COLUMN_MAJOR means that all columns in a row are searched before moving to the next row, and vlistFIND_ROW_MAJOR means all rows are checked before moving to the next column (refer to Figure). When the match is found, its location is returned through row and column and TRUE is returned. If no match is found then 0 is placed in row and column, and FALSE is returned.

vlistREMOVE_RANGE

        <vlistREMOVE_RANGE>
Removes a range of rows or columns from list. All rows after the removed range are moved up to close up the empty space. If the target row does not exist, or if insufficient rows exist to be removed, and exception is raised. All observers are notified of the change. All selections that are not immutable (vlistSetSelectionImmutable) are adjusted to have the same cells selected after the removal. victim is the row or column to start the removal from, and count is the number of rows or columns to remove. type is vlistROW or vlistCOLUMN to indicate if this is a row or column operation. If you are using a custom data representation, this method should be overridden to provide removal functionality. You should always call the superclass's implementation so that observers will be notified and selections will be adjusted. If you have indicated to list that you are using a custom data representation (by calling vlistSetCellValueGetter, vlistSetCellValueSetter, or vlistSetDataset), the default implementation of this will not attempt to manipulate the data.

vlistSET_ORIENTATION

        <vlistSET_ORIENTATION>
Changes the data storage orientation for list to be direction. Values for direction can be vlistROW_MAJOR or vlistCOLUMN_MAJOR.

The orientation is a hint to the List Manager that manipulation operations in a particular dimension are more prevalent than in the other. For example, a tabular database could have row operations (moving, inserting, sorting) occur more frequently than the same operations on the columns. In this case, the list orientation should be set to vlistROW_MAJOR. The default orientation is vlistCOLUMN_MAJOR.

Note, for one-dimensional lists (one row or one column) that it is unwise to set the orientation to be larger dimension (vlistROW_MAJOR for a one-column list). This will not increase performance and will double the list storage overhead.

List orientation also affects how list will store itself in a resource, so you should always call the superclass's implementation so that observers will be notified. If you have indicated to list that you are using a custom data representation (by calling vlistSetCellValueGetter, vlistSetCellValueSetter, or vlistSetDataset), the default implementation of this will not attempt to reorient the the data.

        <vlistSET_ROW_COLUMN_COUNT>

When rows are removed, the client is responsible for making sure any memory allocated for the cell values is released. type is vlistROW to change the row count, and vlistCOLUMN to change to column count.

vlistSORT

        <vlistSORT>
Sorts the list using the cell contents. axis denotes the part of the list to exchange during sorting, if axis is vlistROW, then rows are exchanged. If axis is vlistCOLUMN, then columns are exchanged. key is the index of the row or column to be used as the key for comparing the rows or columns. If axis is vlistROW, then key denotes the column that contains the keys; if axis is vlistCOLUMN, then key denotes the row that contains the sort keys.

The compare function is called with the two values returned by the CellValueGetter for the cells being compared, it should return a value less that zero if val1 < val2, a value greater than 0 if val 1 > val2, and zero if val1 == val2. The values passed to the sort function are the cell contents of the key field. Passing NULL for compare calls a default comparison function (vcharCompareBase).

The sort used by vlistSORT is a merge sort, which is a stable sort with good performance characteristics for sorted and unsorted data.

vlistSORT_BY_NAME

        <vlistSORT_BY_NAME>
Sorts the list using the cell names rather than the cell contents. axis denotes the part of the list to exchange during sorting, if axis is vlistROW, then rows are exchanged. If axis is vlistCOLUMN, then columns are exchanged. key is the index of the row or column to be used as the key for comparing the rows or columns. If axis is vlistROW, then key denotes the column that contains the keys; if axis is vlistCOLUMN, then key denotes the row that contains the sort keys.

The compare function is called with the two cell names returned by the cell value namer procedure for the cells being compared, it should return a value less that zero if val1 < val2, a value greater than 0 if val 1 > val2, and zero if val1 == val2. The values passed to the sort function are the cell contents of the key field. Passing NULL for compare calls a default comparison function (vcharCompareBase).

The sort used by vlistSORT_BY_NAME is a merge sort, which is a stable sort with good performance characteristics for sorted and unsorted data.

vlistSTORE_CELL

        <vlistSTORE_CELL>
Writes the contents of the cell (row, column) of list into the resource resource. resource should be a resource created by vlistMakeMinorResource. The cell value can be recovered from the resource at a later time with vlistLoadCell. The default implementation of vlistStoreCell does nothing, but exists as a placeholder for subclasses to add storage features, so it is necessary to override this method to perform storing of list contents. It is not necessary to call the superclass when overriding this method.

vlistSWAP_RANGE

        <vlistSWAP_RANGE>
Exchanges the cell contents of two ranges of rows or columns. One range starts with row/column swap1, and the second range starts with row/column swap2. count indicates the number of rows or columns in each range, and type is vlistROW or vlistCOLUMN to indicate a row or column operation. All observers are notified of the change. All selections that are not immutable (vlistSetSelectionImmutable) are adjust to have the same cells selected after the swap. Swapping behavior is undefined if the two ranges overlap. If you are using a custom data representation, this method should be overridden to provide swapping functionality. You should always call the superclass's implementation so that observers will be notified and selections will be adjusted. If you have indicated to list that you are using a custom data representation (by calling vlistSetCellValueGetter, vlistSetCellValueSetter, or vlistSetDataset), the default implementation of this will not attempt to manipulate the data.

vlistSWAP_ROW_COLUMN
