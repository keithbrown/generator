vexClass

        typedef ... vexClass;
Defines an exception class. When defining a new exception type, you must define the parent type, constructor and destructor, and name of the type. These attributes, combined with an optional dump procedure, form a ``class.'' The definition of a class is stored in a vexClass. There is one root class from which all others are descended (refer to vexGetRootClass).

NOTE:

Exception ``classes'' are C++ classes.

Defines a pointer to a function that, for the exception exception, prints the data specific to a type in the inheritance tree. For example, to dump an /Arg/Range exception, first the dumper for /Arg/Range is called, then the dumper for /Arg, and finally, the dumper for / itself would be called.

vexException

        typedef ... vexException;
Describes a single exception record that is created during an exception generation call and then passed to the handlers as information. It represents one aspect of an exception chain and stores the file name, procedure name, and line number where this exception occurred. It may optionally contain a message string (meant for users to read) describing the cause of the exception. There can also be exception specific data in vexException.

vexArgPOSITION

        void vexArgPOSITION(
                void * exception,
                va_list * argList);
Indicates that the next argument in an exception generation call is an int representing the position of the argument that caused the exception in its argument list. The position is 1-based because 0 indicates an unknown position. In C++ the position number includes any implicit self argument that is passed so a 2 represents the first explicitly passed arg in most C++ Arg exceptions.

vexClear

        void vexClear();
Resets the current exception chain to the state it was in when the current handler was entered. As a result, this call can only be used in the handler block of a vexWITH_HANDLING or vexWITH_MATCHING construct. It may not be used in the try block of any construct. The only reason to use vexClear is when doing a continue, return, goto, or other non-linear branch that crosses the boundaries of an exception construct. If the flow of control falls through to the end of a vexON_EXCEPTION handler, an implicit vexClear is automatically performed for you. vexClear may not be used anywhere in a vexUNWIND_PROTECT construct. Note: because vexClear resets the exception stack to the state it was in at entry into this (the innermost) frame, it cannot be used to break or return out of nested exception handlers. See also vexPopHandler.

vexGenerate

        void vexGenerate(
                const vexClass * clas,
                vexTag tag,
                ... arg3);
Generates an exception and branches to the next handler to handle it. This call never returns; instead, it passes control flow to the last registered handler (or the top handler). This handler can then proceed as it sees fit. vexGenerate takes a variable number of arguments in the form of a NULL-terminated list of key:value pairs, for example each vexTag parameter has a corresponding value parameter. Unless overridden by vexModuleNAME and/or vexLINE, vexGenerate automatically figures out the file and line of the generation call and the value of errno at that moment.

For example: vexGenerate( vexGetArgClass(), vexArgPOSITION, 3, vexMESSAGE, "this argument is too big", vexNULL );

vexPopHandler

        void vexPopHandler();
Pops the topmost handler off of the vex handler stack. This can be used in the try-block of any handler to perform a non-linear break or return that will branch out of the try-block. Each pop will pop one handler off the stack so vexPopHandler must be called once for each nesting level of a multiply nested procedure.

vexUNWIND_PROTECT vexWITH_HANDLING if ((foo()) vexPopHandler(); vexPopHandler(); return; vexON_EXCEPTION bar(); vexEND_HANDLING; vexON_UNWIND baz(); vexEND_UNWIND;

vexPopHandler should not be called from anywhere that is not syntacticly inside at least one try-block.

vexPopHandler is used in situations similar to those where vexClear is used, but vexClear should only be used from a handler block whereas vexClear should only be used from a try-block.

vexPropagate

        void vexPropagate(
                const vexClass * clas,
                vexTag tag,
                ... arg3);
Prepends an exception created from the given information to the current exception chain and branches to the next handler on the list. vexPropagate is intended for use by exception handlers and is only valid inside a handler block. If clas is NULL then the next handler is branched to without another exception being added to the chain. In all other respects this call is equivalent to vexGenerate.

vexUNWIND_PROTECT

        vexUNWIND_PROTECT
Defines a local exception handler, pushes it on to the handler stack, and saves the program state (using either the setjmp ANSI library call or the try reserved word). This is similar to vexWITH_HANDLING except that the exception handler block is executed even if the try block succeeds and at the end of the handler block if there was an exception in this try block it will automaticly be propagated. vexClear may not be used in a vexUNWIND construct.

See also vexON_EXCEPTION and vexEND_UNWIND.

vexWITH_HANDLING

        vexWITH_HANDLING
Defines a local exception handler, pushes it on to the handler stack, and saves the program state using the setjmp ANSI library call.

See also vexON_EXCEPTION and vexEND_HANDLING.

vexWITH_MATCHING

        vexWITH_MATCHING
Begins a local exception matching scope. If no exception is generated before the first vexMATCH directive, control continues after the vexEND_MATCHING directive. If an exception is generated, the control jumps to the first matching vexMATCH directive. If there are no matching directives, control jumps out of the current matching scope to the next exception handling or matching scope above.
