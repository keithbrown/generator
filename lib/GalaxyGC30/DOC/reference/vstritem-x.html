vstritem

        typedef ... vstritem;
A string item is a limited text item object. It is a convenient object to use when you need to display a single line of text, or when you need to receive a small amount of text from the user. When a string item is created, a text object (vstritemText) is created with it; when a string item is destroyed, its text object is destroyed also. A string item is a convenience object for those who need text item functionality, but do not need to customize the text object associated with the text item.

A string item has a set of attributes that configure it's behavior. For example, the maximum length of the string item's content can be specified with vstritemSetMaxLength. Any attempt by the user to input a string that is longer than the specified maximum will be ignored and considered invalid input. If, furthermore, the item was configured to emit an audible error tone when the item's input is considered invalid the input is not only be ignored, but the default system error tone is generated. Refer to the section vstritemSetBeepIfInvalid for more information.

vstritemText

        typedef ... vstritemText;
Contains method and variable bindings for objects of type vstritemText. Classes declared with this type contain the same set of methods and variables as the default string item text data class, but the behavior of methods and the contents of variables may vary. This class type is derived from vtextitemText.

Instances of this type represent text data which can be manipulated via the vtextINSERT_TEXT and vtextDELETE_TEXT methods. An object of this type is manipulated by the vstritem class to provide the features detailed in this document.

vstritemGetNotify

        vstritemNoteProc vstritemGetNotify(
                vstritem * item);
The notification procedure associated with a string item is invoked in response to well defined events that occur on or to an item. These events include:
  1. The item acquires or loses the keyboard input focus. Refer to sections vdialogFOCUS_ITEM and vdialogUNFOCUS_ITEM.
  2. The item's underlying text object is modified by calls to its vtextDELETE or vTextINSERT methods.
  3. vdialogTEST_UNFOCUS_ITEM is invoked on the item and the ValidateOnChange feature is not enabled. Refer to the section vstritemSET_VALIDATE_ON_CHANGE.

The signature of the vstritemNoteProc includes a pointer to the string item instance and a validity code, with the possible values listed below.

  • vstritemNOTIFY_VALID
  • vstritemNOTIFY_PARTIALLY_VALID
  • vstritemNOTIFY_INVALID
Note that these validity codes match those of vformat and, in fact, are mapped directly from the result of invoking vformatDETERMINE_VALIDITY on the format.

If an item's ValidateOnChange feature is enabled and it has a non-NULL format, then, when a modification is attempted against the item's text data, (by user keyboard input or programatically by a call to vstritemSET_STRING), a string is generated. The strings content is the same as if the text data modifying event fully executed.

In other words, the string item can intercept changes to its string content and queries a vformat to determine the validity state of that proposed input. The resulting information is expressed by the item's notification procedure where it is used to keep the end-user informed about the state of the input process. For example, a vdialogLabelItem associated with the string item might change colors, from red for invalid, to yellow for partially valid, and finally to blue for a valid input state.

vstritemGetTransientString

        vstr * vstritemGetTransientString(
                vstritem * item);
Returns a string that is a copy of the shadow text data maintained by the string item for validation purposes. One of the primary features of the string item class is the ability to determine the validity of a candidate content string before allowing the actual modifying operation to actually occur on the item's real text data. This feature is commonly referred to as key- stroke level validation. This feat is accomplished with the help of a shadow text data object that is used to obtain a version of the item's content string as if the modifying event (insertion/deletion) were allowed to go through. From the shadow text data we can conjer up a string which can then be passed of to an type-specific instance of vformat for validity testing. Refer to vstritemSetFormat

The result of invoking this method is a copy of the current contents of the shadow text data, which will either be exactly the same as the item's actual content string or what it will shortly become if the item's NeverInvalid feature is disabled, which it is by default. Refer to vstritemSetNeverInvalid.

This method is useful to string item subclasses that do not make use of a type-specific vformat instance for determining the validity of a candidate content string, but rather, use a customer validation procedure. Refer to the section vstritemSetValidator.

vstritemHasEmptyIsNull

        vbool vstritemHasEmptyIsNull(
                vstritem * item);
Returns an indication that the string item content is to be considered unset, or undefined, when there is no content. This feature is in support of database management systems and arises from the distinction between an empty value and an undefined value. Consider a type- specific derivative of vstritem that supports the input of double values, vdoubleitem. If such a double item were in a database form GUI, and the user left that field blank, perhaps because it was optional, what would be the value of that field? Would it be 0? If so, why? The fact is that any column of a relational base table can be configured to have a default value, and one of those default values may be the null-value. This feature is used to make as seemless as possible the integration between a form-based GUI and an RDBMS.

When a string item has this feature enabled, and the item's string content is empty (has no characters), then any other intrinsic value that the item may represent is to be considered undefined, unset, or random. For example, to continue with the double item, if it is empty, a call to vdoubleitemGetValue will return some random value, but vdoubleitemIsNull will return vTRUE so that random value MUST be ignored and can only be considered the item's last non-NULL value.

vstritemHasValidateOnChange

        vbool vstritemHasValidateOnChange(
                vstritem * item);
Returns an indication that the string item will validate any proposed changes to its string content as they occur. Keystroke-level validation results from enabling this feature. The string item intercepts modifying events, (insertions/deletions) and first carries out the modification on a shadow copy of the item's underlying text data. Next, the vstritemValidate method is invoked, wherein either the instance validation procedure is called or a candidate content string is generated from the item's shadow text (Refer to the section vtextScribeText) and used in a call to vformatDetermineValidity.

To disallow any changes to the item's content string that would lead to an invalid state, Refer to the section vstritemSetNeverInvalid. To cause the item to generate an audible tone whenever the content string is invalid, refer to the section vstritemSetBeepIfInvalid.

vstritemSetNotify

        void vstritemSetNotify(
                vstritem * item,
                vstritemNoteProc notify);
The notification procedure associated with a string item is invoked in response to well defined events that occur on or to an item. These events include instances when:
  1. The item acquires or loses the keyboard input focus (Refer to the section vdialogFOCUS_ITEM and vdialogUNFOCUS_ITEM),
  2. The item's underlying text object is modified (Refer to the section vtextDELETE and vtextINSERT)
  3. vdialogTEST_UNFOCUS_ITEM is invoked on the item and the ValidateOnChange feature is not enabled (Refer to the section vstritemSET_VALIDATE_ON_CHANGE).

The signature of the vstritemNoteProc includes a pointer to the string item instance and a validity code, whose possible values include:

  • vstritemNOTIFY_VALID
  • vstritemNOTIFY_PARTIALLY_VALID
  • vstritemNOTIFY_INVALID

Please note that these validity codes match those of vformat and, in fact, are mapped directly from the result of invoking vformatDETERMINE_VALIDITY on the format. If an item's ValidateOnChange feature is enabled and it has a non-NULL format, when a modification is attempted against the item's text data, via user keyboard input or programatically via a call to vstritemSET_STRING, a string is generated whose content is that which would be observed if the text data modifying event should be allowed to fully occur. In other words, the string item can intercept changes to its string content and asks a vformat to determine the validity state of that proposed input. The resulting information can be expressed back through the item's notification procedure where it can be used to keep the end-user informed about the state of the input process. For instance, a vdialogLabelItem associated with the string item might change colors, from red for invalid, to yellow for partially valid, and finally to blue for a valid input state.

vstritemSetNull

        void vstritemSetNull(
                vstritem * item,
                vbool isNull);
The concept of a Null value is in support of database management systems and implies that the information the item is meant to represent is not defined by this item. Some types have a natural symbol that indicates no value, such as a NULL pointer to a string as opposed to a pointer to an empty string. Other types, such as the integral types, do not have such a special symbol.

The string item classes are meant to bridge, or be adapters between, intrinsic types and text-input items and therefore must be as expressive as the type systems with which they must integrate. RDBMS support the concept of undefined, or NULL, values and therefore so does the string item class. Most of the type-specific derivatives of vstritem] have their EmptyIsNull attribute enabled. If isNull is vTRUE, any intrinsic value the item is meant to represent should be considered undefined.

vstritemValidate

        int vstritemValidate(
                vstritem * item);
Determine the validity state of the string item's content string. If the item has a non-NULL validation procedure, it will be invoked and its result used as the return value for this method. If the item's validation procedure is NULL and the item's format is non-NULL, a string is generated from the item's shadow text data and passed in a call to vformat. Refer to the section vstritem. The possible return values include:
  • vstritemNOTIFY_VALID
  • vstritemNOTIFY_PARTIALLY_VALID
  • vstritemNOTIFY_INVALID

This method is called whenever the validity state of the item's content string is needed. If the item's ValidateOnChange feature is enabled, every time the item is modified the proposed string content is first validated. Also, if the ValidateOnChange feature is disabled, when the TestUnfocus method is invoked on the item, indicating that some other object wants to the keyboard input focus, the content string is validated.

vstritemSET_INFORMING_FIELD

        <vstritemSET_INFORMING_FIELD>
Calls on the string item to set its internal state indicating that it is informing, or telling, its associated field that it has a new value. Each string item can be associated with a single vfield object for which the item acts as a proxy view. As the user manipulates the text representation of the field's intrinsic value, the item informs the field of its new value, but first the item sets its IsInformingField attribute to vTRUE. After informing the field of its new value, the item set its IsInformingField attribute is set back to vFALSE.

This internal state is kept so that circular change- observation loops are not created. For example, a long integer item vlongitem won't update its string content in its vlongitemSetValue method if it determines that it got there as a result of it informing a field of its new value. Refer to vstritemIsInformingField for more information.
